# 当前原型总结与需求对齐

## 1. 结论先行

从当前仓库内容看，系统已经有“原型”，但这些原型还不是可运行代码原型，而是三类文档原型：

1. 产品题目原型：定义了要解决什么问题，以及怎么被评分
2. AI 协作原型：定义了开发过程中 Prompt 如何留痕
3. 工程规范原型：定义了 Git Commit Message 如何标准化生成

换句话说，当前仓库已经把“做什么”“怎么协作”“怎么沉淀工程产出”这三件事先定义出来了，只差把它们进一步落成可运行系统。

## 2. 当前原型盘点

### 2.1 产品题目原型

- 文件：[score_rules/rules.md](../score_rules/rules.md)
- 原型类型：产品原型 / 需求原型
- 作用：定义业务目标、交付形式、评分标准

这是整个系统的核心原型。它没有给具体交互稿和接口协议，但已经明确了产品的双核心任务：

1. 传播链还原
2. 内容核查

它同时给了一个非常重要的外部约束：最终不是交论文，而是要在复试现场做 15 分钟产品演示和 15 分钟实现讲解。因此，这个原型天然要求系统既要能跑，又要好讲、好展示。

### 2.2 Git Commit 规范原型

- 文件：[score_rules/git_commit_rules.md](../score_rules/git_commit_rules.md)
- 原型类型：工程协作原型
- 作用：约束 AI 如何基于 `git diff` 生成中文 Conventional Commits

这个原型不直接参与新闻核查功能，但它在工程层面很重要。它说明整个项目不只是想“做出一个 Demo”，而是希望保留清晰、可追踪的研发过程。

### 2.3 Prompt 回执原型

- 文件：[prompt_receipt/receipt_rules.md](../prompt_receipt/receipt_rules.md)
- 原型类型：AI 研发流程原型
- 作用：当指令以 `[Log]` 开头时，自动记录本次 Prompt 的背景、目标和 AI 处理策略

这个原型定义的是一套面向 AI 协作的可追溯机制。它的意义在于：

- 把 Prompt 设计过程显性化
- 让后续复盘知道每次指令到底想解决什么问题
- 为“AI 原生思维与 Prompt”这一评分项提供过程证据

## 3. 各原型的实现逻辑

## 3.1 产品题目原型的实现逻辑

这个原型背后的逻辑不是“做一个新闻摘要器”，而是“做一个证据驱动的新闻分析工作流”。

它隐含的业务逻辑可以拆成以下主链路：

1. 用户提供一条新闻
2. 系统识别这是一个什么事件
3. 系统去追踪该事件在外部世界里的传播过程
4. 系统把原文拆成可验证的 claim
5. 系统为每个 claim 检索证据并作出判断
6. 系统生成适合展示的结构化报告

这个逻辑说明系统不应该是“一次大模型生成全部答案”，而应该是一个多阶段 pipeline。

### 这个原型更像什么

更像：

- 事件分析系统
- 证据驱动核查系统
- 新闻观察员工作台

不像：

- 简单聊天机器人
- 单纯摘要工具
- 只给真假标签的二分类器

## 3.2 Git Commit 规范原型的实现逻辑

这个原型的实现逻辑非常明确：

1. 输入 `git diff`
2. 提炼出变更主题
3. 识别变更属于哪一类 `type`
4. 输出符合 Conventional Commits 的中文提交信息

它本质上是一个“结构化代码变更解释器”。它要求 AI 不输出多余说明，而只输出规范化结果。

这反映出的实现思路是：

- 输入是半结构化变更上下文
- 处理阶段是语义归纳与分类
- 输出是严格格式化文本

这个逻辑对后续主系统也有借鉴意义，因为新闻核查同样需要：

- 先做信息抽取
- 再做分类和归纳
- 最后输出结构化结果

## 3.3 Prompt 回执原型的实现逻辑

这个原型的逻辑是一个典型的“触发式记录系统”：

1. 当用户输入符合触发条件时
2. AI 先完成主任务
3. 再把本次任务目标、上下文和策略记录到日志文件

它的核心不在功能复杂度，而在流程约束。它告诉我们，这个项目的开发方式不是一次性完成，而是希望留下决策链路和 Prompt 迭代痕迹。

这对后续产品实现非常重要，因为题目本身就鼓励“充分利用 AI 工具”，而 Prompt 日志就是最好的过程证明材料之一。

## 4. 各原型的实现思路

## 4.1 产品题目原型的实现思路

基于题目原型，当前最合理的实现思路应当是“检索增强 + LLM 分阶段处理 + 可视化展示”。

原因如下：

- 传播链还原必须依赖外部信息源，不能只靠大模型记忆
- 内容核查必须以证据为基础，不能只靠生成模型自由判断
- 现场演示需要直观的 UI，说明前端展示不能缺位

因此它最自然的实现思路是：

1. 输入标准化
2. 事件理解
3. 传播链检索与重建
4. claim 抽取与核查
5. 结果报告生成
6. 前端展示

## 4.2 Git Commit 规范原型的实现思路

这个原型的实现思路是“规则优先，生成次之”。

原因是 Commit Message 有明确格式约束，所以不需要复杂模型推理链，而是需要：

- 清晰的类型集合
- 明确的输出模板
- 对变更范围和意图的压缩表达

如果后面要把它做成工具能力，可以直接作为：

- 本地 CLI 助手
- Git Hook 辅助命令
- IDE / Agent 内置 Prompt 模板

## 4.3 Prompt 回执原型的实现思路

这个原型的实现思路是“事件触发 + Markdown 追加记录”。

它不强调复杂算法，而强调过程一致性：

- 什么时候记
- 记什么
- 用什么统一格式记

如果后续扩展，可以进一步演化成：

- Prompt 操作审计日志
- AI 协作日报素材
- Prompt 复盘库

## 5. 各原型隐含的实现技术

## 5.1 产品题目原型隐含的技术栈

虽然题目没有强制技术选型，但从任务结构看，它天然指向以下技术能力：

- Web 或可视化前端
- 后端任务编排服务
- LLM 接入层
- 搜索 / 检索接口
- 网页正文抽取
- 结构化输出 schema
- 缓存与错误处理

如果要在有限时间内做出稳定版本，技术上最稳妥的组合一般会是：

- 前端：React / Next.js
- 后端：FastAPI
- 分析编排：Python
- 数据层：SQLite / 本地缓存 / JSON 文件缓存
- AI 层：LLM + Prompt 模板 + Schema 校验

## 5.2 Git Commit 规范原型隐含的技术实现

这个原型技术上很轻，但很明确：

- 输入依赖：`git diff`
- 处理方式：Prompt 模板 + 变更语义归纳
- 输出介质：纯文本
- 适配场景：CLI、IDE Agent、Git Hook

它体现的是“语言模型做受约束生成”的思路。

## 5.3 Prompt 回执原型隐含的技术实现

这个原型需要的技术点也很简单，但非常实用：

- 指令触发识别
- 文件是否存在检查
- Markdown 内容拼接
- 日志格式模板化

这说明项目后续很适合建立一个统一的“AI 工作流层”，把日志、提交信息、分析任务都纳入同一套 Agent 协作机制。

## 6. 结合你的需求后的统一理解

你当前的需求不是继续写散的分析，而是基于“现有原型”形成一个统一认知：

- 当前到底已经有什么
- 这些原型分别解决什么问题
- 它们将来在系统里应该处于什么位置
- 后面真正开发时，应该怎样把这些原型串起来

结合这一点，可以把当前系统理解成三层：

### 6.1 业务产品层

对应 `score_rules/rules.md`。

负责回答：

- 产品做什么
- 核心能力是什么
- 评分看什么

### 6.2 AI 编排层

对应未来真正要落地的新闻分析系统。

负责回答：

- 输入如何被理解
- 检索如何展开
- claim 如何抽取和核查
- 报告如何生成

### 6.3 研发治理层

对应 `git_commit_rules.md` 和 `prompt_receipt/receipt_rules.md`。

负责回答：

- 与 AI 协作的过程如何被记录
- 研发变更如何被规范化沉淀

也就是说，当前仓库的原型不是彼此无关，而是已经隐含了一套完整的项目方法论：

- 用产品原型定义业务目标
- 用 AI 工作流原型定义协作方式
- 用工程规范原型定义交付质量

## 7. 从原型到可运行系统，还缺什么

当前原型已经定义了方向，但距离可运行版本还缺几类关键实体。

## 7.1 缺少代码级原型

现在还没有：

- 前端页面原型
- API 原型
- 后端模块骨架
- 数据结构定义

这意味着下一步不能再停留在概念层，必须开始补：

- 输入输出 schema
- API 路由
- 服务模块划分
- 页面布局草图

## 7.2 缺少传播链的数据来源方案

传播链是最容易说得漂亮、做得空的模块。当前原型定义了目标，但还没有明确：

- 用什么搜索源
- 如何拿到相关新闻时间线
- 如何做去重和关键节点识别

这部分一旦不尽快定方案，后面实现会发散。

## 7.3 缺少 claim 级核查协议

目前还没有明确：

- claim 抽取后的数据结构
- evidence 的字段设计
- verdict 的标签集合
- confidence 的定义方法

这会直接影响后端和前端是否能顺利联动。

## 7.4 缺少演示用样例与缓存机制

题目是复试演示导向，所以技术方案不能只追求“理论上可跑”，还要考虑：

- 现场网络抖动
- 第三方接口不稳定
- Demo case 是否稳定复现

因此缓存和样例库其实是主系统的一部分，不是最后补的小细节。

## 8. 对后续实现最有价值的归纳

如果把当前所有原型浓缩成几个关键结论，最重要的是下面这些：

1. 当前仓库已有的是“文档原型”，不是“代码原型”
2. 主系统应该是一个证据驱动的新闻分析 pipeline，而不是一个单轮问答机器人
3. 传播链和内容核查是两个并行核心模块，不能只做其中一个
4. Prompt 日志和 Commit 规范不是附属品，而是项目工程化的一部分
5. 下一阶段最该补的是 API、schema、页面原型和模块骨架

## 9. 推荐的下一步

基于现有原型，后续最合理的推进顺序应该是：

1. 先补“系统蓝图”
   - 明确 API、核心数据结构、页面模块
2. 再补“最小可跑骨架”
   - 前端输入页 + 后端分析接口 + 假数据报告页
3. 然后补“真实能力”
   - 传播链检索
   - claim 抽取
   - evidence 检索
   - verdict 生成
4. 最后补“演示稳定性”
   - 缓存
   - 样例
   - 错误兜底
   - README 和讲解材料

## 10. 最终判断

当前系统的原型已经足够支撑我们进入真正实现阶段了，因为它已经明确了三件最关键的事：

- 产品要解决什么问题
- AI 协作过程如何沉淀
- 工程输出如何标准化

所以接下来的重点不应该继续扩大需求讨论，而应该进入“把原型翻译成代码结构”的阶段。
