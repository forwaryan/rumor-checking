# 高分导向的缺口分析与规则补强建议

## 1. 先说结论

当前仓库已经把三件事定义得比较清楚：

1. 题目要做什么
2. V1 产品大概该怎么做
3. AI 协作和 Commit 如何留痕

但如果目标是复试拿高分，现阶段还缺一层更关键的内容：

- 评分项到执行项的映射规则
- Demo 稳定性与答辩稳定性的规则
- 证据质量、判定边界、异常处理的规则
- 工程验收和 README 交付标准

简单说，现在的文档更像“方向是对的”，还不够“怎么稳拿分”。

## 2. 当前已有内容的优点

### 2.1 题目理解基本正确

- 已经明确这是“双核心任务”：传播链还原 + 内容核查
- 已经明确不能做成单轮问答机器人，而应做成证据驱动 pipeline
- 已经明确 Web App 更适合演示和得分

这些判断与题目评分点是对齐的。

### 2.2 V1 范围控制是合理的

- 没有盲目追求全网爬虫、复杂图谱、多模态
- 选择了 `claim -> evidence -> verdict` 的经典 fact-checking 路线
- 已经开始考虑缓存、降级、demo case

这对“核心功能完整性”和“实现方法”是加分的。

### 2.3 已有 AI 协作意识

- 有 Prompt 留痕规则
- 有 Git Commit 规范
- 有开源参考与方法借鉴

这部分能支撑“AI 原生思维与 Prompt”“实现方法”的讲解。

## 3. 当前最明显的缺口

## 3.1 缺少“评分标准 -> 验收标准”的明确映射

题目给了评分维度和权重，但仓库里还没有把它们进一步翻译成可执行的 DoD（Definition of Done）。

这会带来两个问题：

- 开发时容易做很多“看起来高级”但不加分的东西
- 答辩时不容易把成果准确映射回评分项

这是当前最该补的一层规则。

## 3.2 缺少“证据质量规则”

虽然已经强调 evidence 很重要，但还没有明确：

- 什么来源算高质量证据
- 官方通报、主流媒体、转载、自媒体的权重怎么区分
- 证据冲突时怎么处理
- 什么时候只能输出 `insufficient`

如果没有这层规则，系统很容易出现“看起来在核查，实际上在拼接网页摘要”的问题。

## 3.3 缺少“传播链判定规则”

当前对传播链的设计是对的，但还不够可执行。还缺少：

- 什么叫起源节点
- 什么叫扩散节点
- 什么叫高峰节点
- 什么叫转折节点
- 引用关系弱、时间相近、内容相似时如何归并

没有这层规则，传播链很容易在答辩时被追问到“你为什么说这里是关键节点”。

## 3.4 缺少“异常与失败场景规则”

现在只有原则性提法，还缺少明确的失败分流：

- 搜索不到结果怎么办
- URL 抽取失败怎么办
- 文章太长导致上下文超限怎么办
- 多条证据互相冲突怎么办
- 模型输出不符合 schema 怎么办

这一块直接影响“核心功能完整性”和“AI 原生思维与 Prompt”。

## 3.5 缺少“Demo 作战规则”

题目明确是现场演示导向，但当前仓库还没有一份专门的 Demo 规则，至少应明确：

- 准备几个固定案例
- 每个案例看什么亮点
- 演示失败时怎么降级
- 15 分钟产品介绍怎么分配
- 15 分钟实现讲解怎么分配

这一点很实际，而且对得分影响极大。

## 3.6 缺少“工程验收规则”

现在有 Commit 规范，但还没有更重要的工程交付规则，例如：

- README 必须包含哪些内容
- 哪些模块必须有最小测试
- 环境变量、配置、缓存目录如何说明
- 错误日志、重试策略、超时策略是否统一

Commit Message 规范是加分项，但不如 README、模块结构、测试和稳定性重要。

## 3.7 Prompt 留痕规则偏轻，支撑评分还不够

当前 `prompt-history.md` 只记录：

- 原始指令摘要
- 核心意图
- AI 采用的策略

它适合留痕，但不够支撑“Prompt 设计精巧、异常情况处理充分”这一评分项。

还缺少：

- 关键 Prompt 版本
- Prompt 解决了什么失败问题
- 输出 schema
- 幻觉防控策略
- 上下文超限处理策略

## 4. 想拿高分，最该注意什么

## 4.1 第一优先级不是“功能更多”，而是“主链路更稳”

最高权重是“核心功能完整性”，所以最重要的是：

- 传播链能出
- claim 核查能出
- 证据能点开
- 整体演示流畅

如果只能二选一，应该优先保证稳定和清晰，而不是追求炫技功能。

## 4.2 一定要敢于输出“不确定”

这个题最容易失分的地方，是模型自信地下结论。

高分策略反而是：

- 证据不够时明确说证据不足
- 观点类不强行判真假
- 冲突证据时明确展示冲突
- 传播链无法完全还原时说明还原范围

这会显著提升系统可信度，也更符合“较真”的产品定位。

## 4.3 结果页必须让人一眼读懂

高分产品体验不等于页面复杂，而是信息层次清楚：

- 先看结论
- 再看传播链
- 再看 claim 表
- 最后看证据细节

如果面试官需要自己想很久才能理解页面结构，这一项会很吃亏。

## 4.4 要主动展示“AI 不是乱生成，而是被约束地工作”

答辩时应该主动讲清楚：

- 哪些步骤是检索
- 哪些步骤是 LLM 抽取/归纳
- 哪些步骤用了 schema 校验
- 哪些场景有 fallback

这比单纯说“我接了某个模型 API”更有说服力。

## 4.5 要准备“为什么这样设计”的答法

高分往往不只看功能能不能跑，还看有没有工程判断力。至少要能解释：

- 为什么不做复杂图谱
- 为什么 claim 要原子化
- 为什么要先证据后 verdict
- 为什么前端用单页闭环
- 为什么加缓存和 demo case

这些问题其实就是评分项里的“产品思维”“工程质量”“实现方法”。

## 5. 建议新增的规则

下面这些规则最值得补，而且优先级高。

## 5.1 新增 `rules/high_score_checklist.md`

作用：把官方评分项翻译成项目验收清单。

建议至少覆盖：

- 产品思维与体验：页面结构、输入路径、结果表达、可解释性
- 核心功能完整性：传播链、claim 核查、证据展示、错误兜底
- 代码与工程质量：目录结构、schema、README、测试、日志
- AI 原生思维与 Prompt：多阶段 pipeline、结构化输出、幻觉防控
- 实现方法：开源复用、架构选择理由、AI 工具协作痕迹

这是最优先应该补的规则。

## 5.2 新增 `rules/evidence_and_verdict_rules.md`

作用：定义核查模块的硬边界，减少“随口判断”。

建议明确：

- evidence 最小字段
- evidence 来源分级
- verdict 标签定义
- confidence 的含义
- 冲突证据处理规则
- `insufficient` 的触发条件

建议 evidence 至少包含：

- `title`
- `url`
- `source_name`
- `published_at`
- `snippet`
- `relevance_reason`
- `source_tier`

## 5.3 新增 `rules/propagation_chain_rules.md`

作用：定义传播链模块如何判定关键节点。

建议明确：

- 起源节点判定规则
- 扩散节点判定规则
- 高峰节点判定规则
- 转折节点判定规则
- 去重与聚类规则
- 引用关系强弱的判定规则

这样传播链结果才更可解释，也更抗追问。

## 5.4 新增 `rules/failure_handling_rules.md`

作用：把“失败场景”从口头建议变成工程规则。

建议明确：

- URL 抽取失败 fallback 到纯文本输入
- 搜索失败时只做本地 claim 分析
- schema 校验失败时自动重试
- 上下文过长时分块摘要
- 外部依赖超时时返回部分结果
- 没有证据时禁止输出确定性 verdict

这份规则会直接拉高稳定性和答辩可信度。

## 5.5 新增 `rules/demo_and_rehearsal_rules.md`

作用：为复试演示服务。

建议明确：

- 至少准备 3 个 demo case
- 每个 case 的亮点、风险、备选演示路径
- 哪些结果允许预缓存
- 现场断网或接口抖动时如何切换
- 产品演示和实现讲解各自的时间分配

这份规则的价值非常高，因为题目是 Demo 导向，不是论文导向。

## 5.6 新增 `rules/engineering_acceptance_rules.md`

作用：把“工程质量”从抽象印象变成可检查项。

建议明确：

- README 必须包含安装、运行、架构、限制、Demo 方法
- 核心 schema 必须显式定义
- Prompt 模板必须单独存放
- 至少有核心 pipeline 的 smoke test
- 缓存、日志、配置项必须有统一约定

## 5.7 新增 `rules/prompt_and_eval_rules.md`

作用：专门服务“AI 原生思维与 Prompt”这一评分项。

建议明确：

- 各阶段 Prompt 的职责边界
- 输出 schema 要求
- hallucination guard 提示词约束
- self-check / cross-check 策略
- Prompt 调优记录格式
- 关键 case 的评测方式

当前的 `prompt-history.md` 更像过程日志，不等于 Prompt 设计规范。

## 6. 可以少花时间的地方

如果目标是高分，不建议现在把时间过多花在这些地方：

- 过度打磨 Commit Message 规则
- 过多扩写泛泛而谈的需求分析
- 做复杂但不稳定的社交传播图
- 做很重的爬虫或知识图谱平台
- 做没有证据引用的“智能总结”

这些内容不是完全没价值，但短期收益不如“规则补强 + 稳定实现 + 演示策略”。

## 7. 最推荐的补强顺序

建议按下面顺序推进：

1. 先补 `high_score_checklist`
2. 再补 `evidence_and_verdict_rules`
3. 再补 `propagation_chain_rules`
4. 再补 `failure_handling_rules`
5. 再补 `demo_and_rehearsal_rules`
6. 最后补 `engineering_acceptance_rules` 和 `prompt_and_eval_rules`

原因很简单：

- 前四项决定产品能不能可信地跑
- 第五项决定现场能不能稳地讲
- 后两项决定能不能把工程和 AI 思维讲得更完整

## 8. 最终判断

当前仓库不是方向错了，而是还停留在“设计思路正确”的阶段。

如果想把分数往上拉，下一步最该做的不是继续泛化分析，而是补出一套面向复试得分的执行规则：

- 怎么算做完
- 什么情况下不能乱判
- 失败时怎么兜底
- 演示时怎么稳
- 工程交付至少达到什么标准

一句话总结：

现在缺的不是“再想一个更大的方案”，而是“把高分路径写成明确规则并落实到实现与演示里”。
