# 实现难点分析与两天落地策略

## 1. 先说结论

当前项目真正的难点，不是“会不会写页面”或者“会不会接大模型”，而是：

1. 如何把一个边界模糊的题目重新定义成两天内可完成的最小闭环
2. 如何在高度依赖外部证据和外部检索的前提下，做出可解释、可演示的结果
3. 如何在“传播链还原”和“内容核查”这两个不同任务之间做 scope 控制

如果目标被定义成：

- 自建新闻数据库
- 自建传播图谱
- 自建完整事实核查平台

那么项目难度会远超两天可交付范围。

如果目标被定义成：

- 借助外部数据源和开源组件
- 面向单条新闻事件
- 输出关键来源时间线和 claim 级核查表

那么项目难度会收敛到一个可执行的验证型系统。

因此，这个项目的关键不是“功能做得越多越好”，而是“边界定得越准越容易赢”。

## 2. 当前最核心的难点在哪里

## 2.1 难点一：题目本身定义不够清晰

题目要求“传播链还原 + 内容核查”，但没有严格规定：

- 什么叫传播链
- 什么粒度才算还原
- 什么样的核查结果才算可信

这会直接导致一个问题：如果不先补定义，后续任何实现都可能在答辩时被追问“为什么这就算传播链”“为什么这样就能得出结论”。

### 为什么这是高风险

- 它影响产品边界
- 它影响技术路线
- 它影响验收口径

### 两天内怎么处理

先主动重新定义：

- 传播链 V1 = 关键来源时间线，不承诺完整社交扩散图
- 内容核查 V1 = 对新闻中的若干原子 claim 做证据判断

只要定义清楚，很多原本不可做的事就会变成可做。

## 2.2 难点二：最难的不是判断，而是找证据

大模型可以很快生成一个“像结论”的回答，但事实核查真正困难的地方在于：

- 去哪里找证据
- 找到的证据是否可信
- 证据之间冲突时怎么处理
- 什么时候应该输出“证据不足”

如果没有证据协议，系统很容易退化成“基于检索结果的新闻摘要器”，而不是核查系统。

### 为什么这是高风险

- 它决定结果是否可信
- 它决定系统是否具备“可解释性”
- 它是最容易在演示时被质疑的地方

### 两天内怎么处理

不要追求“所有结论都能判断”，而要强制支持以下输出：

- `supported`
- `refuted`
- `insufficient`
- `conflicting`

重点不是多判断，而是敢于承认不确定。

## 2.3 难点三：中文新闻场景的输入质量不稳定

这个项目默认处理的是中文新闻语境，但中文新闻网页存在几个现实问题：

- 页面结构不统一
- 发布时间字段不稳定
- 有转载、搬运、改写
- 有些页面对爬取不友好
- 正文、摘要、引用内容可能混在一起

这意味着系统的第一步“拿到干净输入”本身就不稳。

### 为什么这是高风险

- 输入不稳，后面所有步骤都会漂
- 时间线排序会直接受发布时间质量影响
- claim 抽取会被噪声文本污染

### 两天内怎么处理

先接受“抽取能力不完美”这个事实，用降级策略控制风险：

- 优先支持粘贴正文
- URL 抽取作为增强能力
- 对抽取失败场景给出明确提示
- demo 只选抽取稳定的样例新闻

## 2.4 难点四：传播链和内容核查其实是两套能力

这两个子任务相关，但并不等价。

传播链偏向：

- 时间顺序
- 来源关系
- 相似报道聚合

内容核查偏向：

- claim 拆分
- 证据检索
- verdict 判定

如果试图在两天内同时把这两块都做“完整”，最常见的结果就是两边都不够稳。

### 为什么这是高风险

- 容易造成架构过重
- 容易把 prompt 和逻辑绕得过深
- 容易在最后一天发现两边都没有完全跑通

### 两天内怎么处理

必须明确主次：

- 主线一：关键来源时间线
- 主线二：claim 级核查表

不要再派生出第三条复杂主线，例如社交传播图、复杂实体图谱、多模态核查。

## 2.5 难点五：外部资源可用，但不完全可控

这个项目如果不自建数据库，就一定要依赖外部资源，例如：

- 新闻抽取库
- 搜索接口
- GDELT
- 事实核查 API
- LLM 接口

这些能力能显著降低开发成本，但也带来新的问题：

- 网络波动
- 限流
- 返回质量不稳定
- 某些新闻搜不到
- 某些 claim 没有现成核查结果

### 为什么这是高风险

- 演示时最容易出故障
- 本地调试和现场展示表现可能不一致
- 一旦外部服务慢，整体链路就会变得不可控

### 两天内怎么处理

设计时就把“缓存”和“降级”考虑进去：

- 先缓存 demo case 的检索结果
- 对找不到证据的情况输出 `insufficient`
- 对接口失败情况返回“检索失败，请改用文本输入或稍后重试”

不要假设网络永远正常。

## 2.6 难点六：缺少统一验收标准，容易越做越散

当前仓库已经有很多设计文档，但仍然缺少最关键的可执行验收口径，例如：

- 时间线至少多少节点才算完成
- claim 至少核查几条才算完成
- 证据链接缺多少还能演示
- 页面结果怎样才算“讲得清楚”

如果没有这层标准，就很容易出现：

- 觉得还差一点，再加一个功能
- 一边做一边改目标
- 最后功能很多，但最核心的主链路不稳

### 为什么这是高风险

- 它会直接拖慢推进速度
- 它会导致实现优先级不断漂移

### 两天内怎么处理

先锁定最低验收标准：

- 支持 1 条新闻输入
- 输出 5 到 10 个关键时间线节点
- 输出 3 到 5 条 claim 核查结果
- 每条结果尽量附来源或证据链接
- 允许一部分结论为 `insufficient`

这套标准要先于开发冻结。

## 2.7 难点七：Demo 稳定性的重要性高于算法先进性

两天项目里，真正决定成败的往往不是算法多先进，而是：

- 能不能稳定演示
- 能不能在几十秒到数分钟内给出结果
- 能不能在结果页上讲清楚“为什么是这个结论”

很多“高级设计”在工程上并不会加分，反而会提高翻车概率。

### 为什么这是高风险

- 现场演示的失败成本极高
- 答辩时更容易追问“为什么这样设计”，而不是“用了什么最先进模型”

### 两天内怎么处理

优先保证：

- 主链路可跑通
- 输出结构可讲清楚
- demo case 已提前缓存和验证

不要把时间花在炫技点上。

## 3. 难点优先级排序

按对交付结果的影响排序，当前最需要优先解决的是：

1. 明确 V1 边界和验收口径
2. 验证证据检索和相关文章检索是否可用
3. 定义 claim 核查输出协议
4. 设计传播链 V1 的最小实现
5. 处理外部依赖失败和演示缓存
6. 再考虑页面观感和扩展亮点

这个顺序不能反过来。先做 UI、先做复杂 agent 编排、先做知识图谱，都会让主任务失焦。

## 4. 哪些不是当前的主要难点

以下内容看起来高级，但在当前两天目标下，不是最主要的矛盾：

- 要不要多 Agent
- 要不要知识图谱
- 要不要向量数据库
- 要不要复杂前端框架
- 要不要训练模型
- 要不要做多模态核查

这些都属于“后续增强项”，而不是当前卡点。

## 5. 两天内最现实的落地策略

## 5.1 目标重定义

把目标改成：

做一个面向单条新闻的 AI 分析工作台，借助外部数据源和开源组件，输出关键来源时间线、claim 级核查表和综合结论。

而不是：

做一个完整、自建底层能力的新闻事实核查平台。

## 5.2 技术策略

优先复用：

- 新闻抽取库
- 外部检索接口
- 已有事实核查结果 API
- 大模型结构化输出能力

自己重点实现：

- 工作流编排
- 结果结构设计
- 页面展示
- 异常处理和演示策略

## 5.3 Demo 策略

提前准备并缓存 3 条最稳的新闻 case：

- 至少 1 条有较充分公开证据
- 至少 1 条存在争议或反转
- 至少 1 条能演示“证据不足”

这样答辩时既能展示能力，也能展示边界意识。

## 6. 建议立即验证的事项

在正式开写系统前，最值得先做的不是搭完整架构，而是做 3 个小验证：

1. 输入验证
   - 选 5 到 10 条新闻，验证 URL 抽取是否稳定
2. 检索验证
   - 验证能否召回同一事件的多篇相关文章
3. 核查验证
   - 验证 claim 能否稳定输出 `supported / refuted / insufficient / conflicting`

只要这三件事验证通过，后面的工程就是拼装和收敛；如果这三件事不通，再漂亮的系统设计也只是空中楼阁。

## 7. 最终判断

当前项目最大的难点，不是“技术不会”，而是：

- 题目边界天然模糊
- 外部证据高度不稳定
- 两个核心任务并不完全同构
- 两天时间迫使我们必须做强力取舍

因此，最优策略不是追求“大而全”，而是主动把问题压缩成一个可验证、可讲解、可稳定演示的最小闭环。

从这个角度看，真正的工程能力不在于“全都自己做”，而在于：

- 识别什么必须自己做
- 识别什么应该借力
- 识别什么现在不该做

这才是当前阶段最重要的实现判断。
