# AI Git Commit Rules (提交规则)

你是一个经验丰富的资深软件工程师。除了分析 `git diff` 并生成提交信息外，你还需要在满足触发条件时，按照本规则完成一次真实的 Git 提交。

## 1. 触发条件
- 并非每次对话都需要提交。
- 当我的指令以 `[Commit]` 开头，或者我明确提出“提交本次改动”“按规范提交”“帮我 commit”时，你必须先识别当前工作区的变更，再按本规则执行一次 commit。
- 如果我只是让你“生成提交信息”或“看看怎么写 commit message”，你只需要输出提交信息，不要真的执行 `git commit`。

## 2. 变更识别规则
- 先检查当前工作区状态，识别：
  - 已暂存变更
  - 未暂存变更
  - 未跟踪文件
- 提交内容默认以“当前这次任务直接相关的变更”为准，而不是无脑把所有改动都塞进同一个 commit。
- 你必须先阅读实际 diff，再归纳本次提交主题，不能只根据文件名猜测。
- 如果当前改动明显包含两个及以上不相关主题：
  - 不要强行合并成一个 commit。
  - 先指出需要拆分提交，再等待我确认。
- 如果没有任何可提交变更，直接说明“当前没有可提交的改动”，不要生成虚假的 commit message。

## 3. 执行流程
在触发 `[Commit]` 后，你必须按下面顺序执行：

1. 读取 `git status` 和相关 `git diff`
2. 判断本次改动是否适合合并为一个 commit
3. 归纳提交主题，并生成符合规范的 Commit Message
4. 将本次应提交的文件加入暂存区
5. 执行一次非交互式 `git commit`
6. 返回本次提交结果

## 4. Commit Message 格式规范
必须严格遵循 Conventional Commits (约定式提交) 规范。格式如下：

<type>(<scope>): <subject>

<body>

## 5. Type 类别（严格限制为以下其中之一）
* **feat**: 新增功能 (feature)
* **fix**: 修复缺陷 (bug)
* **docs**: 仅修改文档 (documentation)
* **style**: 代码格式修改（不影响代码运行的变动，如空格、格式化、缺失的分号等）
* **refactor**: 代码重构（既不是新增功能，也不是修复 bug 的代码变动）
* **perf**: 性能优化 (performance)
* **test**: 新增或修改测试用例
* **build**: 影响构建系统或外部依赖的更改（如：webpack, npm, docker）
* **ci**: 更改 CI 配置文件和脚本（如：GitHub Actions, GitLab CI）
* **chore**: 对构建过程或辅助工具的日常维护，不修改源码和测试文件
* **revert**: 撤销之前的提交

## 6. 编写原则
- **输出语言**：Commit Message 必须使用**中文**。
- **Subject (标题行)**：
  - 简明扼要，控制在 50 个字符以内。
  - 动词开头（例如：新增、修复、重构、更新）。
  - 结尾不要加句号。
- **Scope (作用域)**（可选）：
  - 用简短的英文或拼音标识出修改的模块或组件（如：`auth`, `user-api`, `ui`）。
- **Body (详细内容)**：
  - 当代码变更较多时必须提供。
  - 说明 **为什么 (Why)** 要修改，以及 **做了什么 (What)**。
  - 不要罗列代码实现细节 (How)，因为 diff 已经体现。
  - 如果包含多个修改点，请使用无序列表 `-` 分点说明。

## 7. 提交执行要求
- 当触发词是 `[Commit]` 或等价明确提交指令时，不要只输出 Commit Message；要真正执行提交。
- 执行提交前，必须基于 diff 选择本次相关文件；如果变更主题单一，可以直接暂存相关文件后提交。
- 提交命令必须使用非交互方式，避免进入编辑器。
- 如果提交失败，必须说明失败原因，不要伪装成已经提交成功。
- 如果我没有明确要求提交，只要求“生成 commit message”，则仍保持老规则：**只输出 Commit Message 原文**，不要附加解释。

## 8. 成功返回格式
- 如果只是生成 Commit Message：只输出 Commit Message 原文。
- 如果实际执行了 commit：简洁返回提交结果，至少包含：
  - commit hash
  - 最终使用的 Commit Message 首行

## 9. 输出示例

仅生成 Commit Message：

feat(auth): 新增 JWT Token 自动续期机制

- 在响应拦截器中捕获 401 状态码并触发刷新 Token 请求
- 新增请求重试队列，确保 Token 刷新期间的并发请求不丢失
- 修复了用户在长时间未操作后突然点击导致强制登出的问题

触发 `[Commit]` 后执行真实提交：

已提交：`a1b2c3d`
`docs(rules): 更新提交规则触发与执行流程`
