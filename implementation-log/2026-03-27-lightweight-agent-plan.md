# 轻量 Agent 化实施任务报告

日期：2026-03-27

## 背景

当前仓库已经具备一部分 Agent 化基础能力，但主流程仍然是固定 Pipeline，而不是受控的 Agent Loop。

关键现状：

- [../backend/app/services/analyze_pipeline.py](../backend/app/services/analyze_pipeline.py)
  - 当前主执行链路，负责标准化输入、检索、问题消歧、follow-up 检索、综合判断和报告生成。
- [../backend/app/services/agent_reasoner.py](../backend/app/services/agent_reasoner.py)
  - 已有以 LLM 为核心的 `resolve_question` 和 `synthesize` 能力，但本质上仍是单次结构化调用。
- [../backend/app/services/retrieval_service.py](../backend/app/services/retrieval_service.py)
  - 已有检索工具能力，但尚未抽象成统一 Tool 协议。
- [../backend/app/api/v1/endpoints/analyze.py](../backend/app/api/v1/endpoints/analyze.py)
  - 已有流式输出基础，但当前输出更偏 Pipeline Trace，而不是 Agent 决策过程。

因此，这次“轻量 Agent 化”的目标不是推翻现有项目，而是在保留现有稳定主干的前提下，引入一层受控的小型调查决策能力。

## 1. 最终实现的目标

### 1.1 目标概念

最终希望实现的是一个“受控的谣言核查调查 Agent”，而不是一个完全开放式的通用 Agent。

它需要具备以下能力：

- 接收模糊问题、URL 或新闻文本输入。
- 先决定要不要澄清问题、补检索或抓取正文。
- 允许在证据不足时自动追加 1 到 2 轮调查。
- 每一步都能说明“为什么这么做”。
- 最终仍然输出当前项目兼容的 `Report`，不破坏已有结果结构。

### 1.2 目标例子

例子 1：

- 用户输入：`最近那个女网红脑出血去世是真的假的？`
- 理想 Agent 行为：
  1. 识别对象不明确。
  2. 先检索候选事件。
  3. 选择最可能的对应事件。
  4. 补检索医院通报、平台声明、辟谣来源。
  5. 在证据足够时输出结论；不足时明确停止并说明原因。

例子 2：

- 用户输入：`这个链接里说某地封城是真的吗？`
- 理想 Agent 行为：
  1. 抓取 URL 内容。
  2. 提取核心 claim。
  3. 查询当地政府、权威媒体和辟谣信息。
  4. 生成 claim 判断和时间线。

## 2. 最终达到的效果

### 2.1 效果目标

轻量 Agent 化后的最终效果不是“更会聊天”，而是“更像一个会推进调查的核查助手”。

预期变化：

- 面对模糊问题时不急于直接给结论，而是先做短计划。
- 当首轮证据不足时，会自动补一轮调查，而不是过早结束。
- 过程可以流式展示为“计划、调用工具、观察、决定是否继续”。
- 最终报告与现有结构兼容，但中间过程更像调查。

### 2.2 预测表现

预测效果 1：

- 当前模式下，如果首轮结果较弱，系统更容易直接停在 `insufficient`。
- 轻量 Agent 化后，系统会先判断“是否值得补搜官方来源”，再决定是否结束。

预测效果 2：

- 当前模式下，用户看到的是固定阶段日志。
- 轻量 Agent 化后，用户看到的是明确的调查动作，例如：
  - `plan: 先定位事件`
  - `tool_call: search_news`
  - `observation: 候选事件过多`
  - `decision: 追加官方检索`
  - `tool_call: fetch_url_content`
  - `finalize: 输出报告`

预测效果 3：

- 对面试和演示而言，产品形态会从“分析报告系统”更接近“调查 Agent”。
- 对真实复杂输入而言，系统会更自然地处理模糊对象、弱证据和冲突证据。

### 2.3 举例

场景 A：

- 输入：`最近那个主播是不是已经去世了`
- 当前模式可能效果：
  - 固定执行首轮检索和 follow-up 检索，输出一个结果，但路径是预设的。
- 轻量 Agent 化后可能效果：
  - 系统先展示“目标对象不明确，先定位事件”。
  - 找到候选事件后再补官方说明。
  - 输出“网传去世不实”或“暂无法确认”。

场景 B：

- 输入：一个带有耸动标题的新闻链接。
- 当前模式可能效果：
  - 直接走 URL 内容抽取和报告输出。
- 轻量 Agent 化后可能效果：
  - 抽取正文后发现 claim 风险较高。
  - 自动再搜地方权威来源。
  - 用更高质量证据完成 verdict。

## 3. 轻量 Agent 化与当前模式对比

### 3.1 列表式对比

| 维度 | 当前模式 | 轻量 Agent 化 |
| --- | --- | --- |
| 执行方式 | 固定 Pipeline | 受控的小型 Agent Loop |
| 决策能力 | 少量硬编码分支 | 能决定是否继续调查 |
| 补检索能力 | 只有预设 follow-up | 可以根据观察结果补 1 到 2 轮 |
| 输入适应性 | 中等 | 更强 |
| 可解释性 | 阶段日志 | 调查计划 + 工具调用 + 决策理由 |
| 稳定性 | 更高 | 稍低 |
| 成本和时延 | 更低 | 更高一些 |
| 可测试性 | 更好 | 更复杂 |
| 演示观感 | 分析系统 | 调查 Agent |
| 后续扩展 | 新能力要继续写死进 Pipeline | 新能力更容易接成 Tool |

### 3.2 当前模式的优点

- 稳定，路径明确，结果更可预测。
- 测试更容易做，回归更清晰。
- 成本更低，调用次数更可控。
- 对现有 Demo、验收和回归最友好。

### 3.3 当前模式的缺点

- 面对模糊问题时上限有限。
- 过程更像“阶段执行”，不像“主动调查”。
- 后续接更多工具时容易继续堆硬编码分支。
- 多轮追问和持续调查能力较弱。

### 3.4 轻量 Agent 化的优点

- 更适合谣言核查这类模糊、开放、证据可能不完整的任务。
- 系统能在证据不足时主动补一步调查。
- 过程更易展示，产品形态更有 Agent 感。
- 后续扩展工具会更顺，架构也更自然。

### 3.5 轻量 Agent 化的缺点

- 路径不再完全固定，稳定性会略降。
- 测试难度提高，不能只测最终结果，还要测中间决策。
- 成本和时延增加。
- 如果边界控制不好，容易做成“看起来像 Agent，但收益不明显”。

### 3.6 什么时候用轻量 Agent 化

适合使用轻量 Agent 化的情况：

- 输入经常比较模糊。
- 需要自动补搜、补证据。
- 需要明显展示“调查过程”。
- 需要为后续 Tool 化扩展铺路。
- 目标包括演示、面试亮点和产品形态升级。

### 3.7 什么时候继续使用当前模式

适合继续使用当前模式的情况：

- 目标是稳定交付和确定性输出。
- 输入大多结构明确。
- 更重视回归测试和固定口径。
- 成本和响应时间约束更强。
- 当前阶段优先把业务逻辑做扎实，而不是扩展 Agent 感。

## 4. 实现轻量 Agent 化需要完成的大的步骤

1. 明确 Agent 的边界、约束和停止条件。
2. 设计统一的 Agent 运行状态模型。
3. 把当前已有业务能力封装成标准化 Tool。
4. 实现受控的 Lightweight Agent Loop。
5. 升级流式事件协议和前端展示方式。
6. 增加测试、评估和回退机制。
7. 补充实施文档、演示文档和过程记录。

## 5. 大步骤细化

### 5.1 明确 Agent 的边界、约束和停止条件

目标：

- 避免把项目误做成开放式通用 Agent。

子任务：

- 定义 Agent 名称和职责，例如 `RumorInvestigationAgent`。
- 限定输入范围：
  - 文本问题
  - URL
  - 新闻文本
- 限定允许动作：
  - 最多 2 轮检索
  - 最多若干 URL 抓取
  - 只允许调用白名单 Tool
- 定义停止条件：
  - 已有足够证据
  - 证据不足但达到轮数上限
  - 问题无法定位到稳定事件
- 明确输出约束：
  - 最终仍输出现有 `Report`

### 5.2 设计统一的 Agent 运行状态模型

目标：

- 让“调查过程”成为一等数据结构，而不是只保留最终报告。

子任务：

- 设计 `AgentRun`
  - `run_id`
  - `goal`
  - `status`
  - `current_step`
- 设计 `AgentStep`
  - `step_type`
  - `status`
  - `reason`
  - `summary`
- 设计 `ToolCall`
  - `tool_name`
  - `tool_input`
  - `tool_output_summary`
- 设计 `Observation`
  - 命中了什么证据
  - 证据够不够
  - 是否继续调查
- 设计与现有 `Report` 的关系
  - 复用 `pipeline_trace`
  - 或新增 `agent_trace`

### 5.3 把当前已有业务能力封装成标准化 Tool

目标：

- 最大化复用现有代码，不重写业务核心。

建议抽象的 Tool：

- `search_news`
  - 封装当前检索服务。
- `resolve_question`
  - 封装问题消歧能力。
- `fetch_url_content`
  - 封装 URL 抽取。
- `extract_claims`
  - 封装 claim extractor。
- `judge_claims`
  - 封装 verdict engine。
- `build_timeline`
  - 封装 timeline builder。
- `finalize_report`
  - 封装 report builder。

子任务：

- 给每个 Tool 定义输入输出 schema。
- 统一 Tool 调用结果格式。
- 统一 Tool 失败错误结构。
- 为每个 Tool 补充可展示摘要。

### 5.4 实现受控的 Lightweight Agent Loop

目标：

- 从固定执行链改为“计划、行动、观察、决定是否继续”的受限小循环。

建议最小流程：

1. `plan`
2. `tool_call`
3. `observation`
4. `decision`
5. `finalize`

子任务：

- 设计 Agent Planner 的输入输出。
- 设计每轮可调用的 Tool 白名单。
- 定义决策逻辑：
  - 是否继续
  - 继续时调用哪个 Tool
  - 何时停止
- 强制轮数限制。
- 失败时回退到旧 Pipeline。

### 5.5 升级流式事件协议和前端展示方式

目标：

- 让用户真正看到调查过程，而不仅仅是后端阶段日志。

后端事件建议：

- `plan`
- `tool_call_started`
- `tool_call_completed`
- `observation`
- `decision`
- `final_report`

前端展示建议：

- 调查步骤面板
- 当前证据池面板
- 当前判断面板
- 最终报告面板

子任务：

- 定义新的流式事件结构。
- 对接当前 `/api/v1/analyze/stream`。
- 前端新增调查过程可视化区域。
- 保留现有最终报告展示逻辑。

### 5.6 增加测试、评估和回退机制

目标：

- 避免 Agent 化后稳定性失控。

子任务：

- Tool 单元测试
- Agent Loop 集成测试
- 停止条件测试
- 异常路径和回退测试
- 和现有样例集做前后对比评估
- 增加配置开关：
  - 开启轻量 Agent
  - 关闭轻量 Agent

### 5.7 补充实施文档、演示文档和过程记录

目标：

- 让这次改造可解释、可复盘、可演示。

子任务：

- 新增轻量 Agent 架构说明文档
- 新增事件流说明文档
- 更新 Demo 脚本
- 更新 Smoke Checklist
- 持续把阶段结论记录到本目录

## 建议的实施顺序

建议按以下顺序推进：

1. 先定义 `AgentRun`、`AgentStep`、`ToolCall` 模型。
2. 把现有检索、消歧、抽取、判定、时间线能力统一封装成 Tool。
3. 新增一个最多 2 轮的 `lightweight_agent_runner`。
4. 把新的 Agent 事件接到流式接口上。
5. 前端增加调查过程面板。
6. 最后补齐测试、文档和 Demo 更新。
