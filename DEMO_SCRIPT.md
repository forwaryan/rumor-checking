# 演示脚本与口播提纲

更新时间：2026-03-13（Asia/Shanghai）

## 1. 这份文档给谁用

这份文档给两类人用：

- 第一次接手这个仓库、需要快速准备 demo 的人
- 面试现场负责口播和操作的人

目标不是把实现讲全，而是把当前已经稳定可演示的内容讲清楚，并且把还没完成的边界说准确。

## 2. 先记住一句话介绍

> 这是一个新闻核查工作台。你给它一段新闻、一个链接或一个问题，它会把消息拆成几个可验证说法，告诉你哪些有证据、哪些还不能下结论，并把关键传播节点按时间线展示出来。

如果时间更短，也可以压缩成两句话：

1. 它不是普通摘要器，而是一个“较真”的新闻观察员。
2. 它当前重点演示两件事：内容核查，以及在证据足够时给出可解释的传播节点。

## 3. 演示前先说明的边界

正式操作前，建议先主动说明这 4 点，后面会更稳：

- 当前最稳定的是三条预设 demo case，不是“任意新闻都能完整还原传播链”。
- 前端会优先调用真实 `POST /api/v1/analyze`；如果后端离线或请求失败，demo case 会回退到本地稳定 payload。
- `complete / partial / safe_mode` 是产品边界设计，不是单纯的成功/失败提示。
- URL 正文抽取、真实开放检索、真实 evidence 驱动 verdict 还没有完全收口，不要把当前系统说成通用新闻事实核查引擎。

## 4. 推荐的 5 到 10 分钟演示顺序

| 时间 | 操作 | 口播重点 |
| --- | --- | --- |
| 0:00 - 0:40 | 打开页面，说明输入方式 | “可以贴文本、链接或直接提问，系统会尽量给结构化结果。” |
| 0:40 - 3:00 | 演示 `expired-yogurt` | 先建立信任：主流程通、结构化结果可读、时间线和 claim 能对上。 |
| 3:00 - 5:30 | 演示 `chemical-odor` | 强调不是所有问题都能二元判断，系统会显式暴露冲突证据。 |
| 5:30 - 7:00 | 演示 `morningstar-layoff` | 强调证据不足时系统会保守，不会伪造完整结论。 |
| 7:00 - 8:30 | 主动讲当前限制 | 先讲没做完什么，再讲下一步为什么是 `Cluster-F / C / D` 的重点。 |
| 8:30 - 10:00 | 预留问答 | 准备回答“能不能查任意新闻”“是否已接真实检索”这类问题。 |

如果现场只有 5 分钟，保留顺序不变，但每个 case 只讲一个核心点：

- `complete_mode` 讲“结构化闭环”
- `partial_mode` 讲“冲突证据与边界”
- `safe_mode` 讲“证据不足也不硬判”

## 5. 三条 demo case 怎么讲

### Case 1：`expired-yogurt` / `complete_mode`

**输入**

```text
3月1日海州市市场监管局通报海州新鲜屋部分酸奶超过保质期，涉事门店已停业整改，目前未发现大规模食物中毒病例。
```

**建议操作顺序**

1. 先看状态条，说明这是 `complete_mode`。
2. 再看事件卡片，讲“一句话总结 + 关键词”。
3. 然后看时间线，点出 `origin` 和 `turn` 两个节点。
4. 最后看 claim 表和证据列表，说明为什么能给出 supported。

**这条 case 的亮点**

- 有明确权威来源，适合先建立演示可信度。
- 时间线至少有 2 个关键节点，能展示“传播链不是一句总结”。
- claim 表里既有 `supported`，也有对 opinion 的保守处理。
- 证据层级能看出官方通报和品牌回应的区别。

**推荐讲法**

> 这条 case 用来演示系统在证据比较完整时会怎么工作。它先把事件总结出来，再把混在一起的说法拆成几条 claim，哪些是监管部门明确支持的，哪些只是评价性语言，就分开处理。时间线这里也不是简单列新闻，而是只挑最值得看的节点，比如监管通报和企业回应。

**不要说过头的地方**

- 不要说“系统已经完成真实互联网检索”。这条结果当前更接近规则链路和稳定 payload 对齐下的完整展示。
- 不要说“所有 opinion 都能被判断真假”。当前 opinion 更多是提示边界，不是强判。
- 不要把两条时间线节点说成“完整传播链还原”，它只是当前稳定可展示的关键节点摘要。

### Case 2：`chemical-odor` / `partial_mode`

**输入**

```text
北城区化工厂夜间异味被居民连续投诉，区生态环境局已经进场核查，但媒体称工厂停产整顿，公司又回应只暂停一条产线。
```

**建议操作顺序**

1. 先看状态条，强调这是 `partial_mode`，不是失败页。
2. 再看时间线，说明从投诉、企业回应到监管核查的推进过程。
3. 然后直接看 claim 表里 `conflicting` 的那一行。
4. 最后看风险提示，说明系统为什么要停在 partial。

**这条 case 的亮点**

- 能直接展示“同一事件有冲突版本”的现实场景。
- 时间线里有 `origin / clarification / turn / peak` 四种节点，适合讲“为什么这里还不能下死结论”。
- claim 表里同时存在 `supported`、`conflicting` 和 `insufficient`，很适合讲边界意识。
- 风险提示会明确提醒不要把单一版本当最终事实。

**推荐讲法**

> 这条 case 适合说明我们不是只会输出一个漂亮结论。现实里经常是居民投诉、企业回应、媒体报道和监管核查同时存在，而且彼此并不一致。这里系统没有硬说“工厂肯定完全停产”或“肯定没有问题”，而是把冲突摆出来，并告诉你哪些部分可以确认、哪些还要保留。

**不要说过头的地方**

- 不要把 `partial_mode` 说成“系统没跑通”。这里恰恰是在演示产品的可信边界。
- 不要说“系统已经验证出污染事实成立”。当前稳定支持的是“投诉存在、监管已进场核查”，不是污染结论已经坐实。
- 不要把媒体报道和公司回应的冲突解释成“系统已经判定谁真谁假”。当前是显式保留冲突。

### Case 3：`morningstar-layoff` / `safe_mode`

**输入**

```text
晨星生物已经宣布裁员40%了吗？
```

**建议操作顺序**

1. 先看状态条，强调这是 `safe_mode`。
2. 再看事件卡片，说明当前只有待核实问题，没有完整事件材料。
3. 接着看 claim 表，指出都是 `insufficient`。
4. 最后看空时间线和风险提示，解释为什么系统选择保守收口。

**这条 case 的亮点**

- 能说明系统不会因为用户问了一个问题就假装已经查实。
- 可以演示 prediction / unverifiable 这类更难直接核验的 claim 类型。
- 空时间线本身就是结果的一部分，代表当前没有足够证据建立传播链。
- 风险提示明确说明“当前页面只适合提示待核查点”。

**推荐讲法**

> 这条 case 主要不是看系统找到了什么，而是看它在找不到可靠证据时怎么停下来。我们宁愿告诉你‘现在不能判断’，也不希望为了演示效果去伪造一个完整结论。对面试场景来说，这条能体现产品边界感和风险意识。

**不要说过头的地方**

- 不要说“safe_mode 代表这条传闻是假的”。它只代表当前证据不足。
- 不要说“系统已经完成全网搜索但没搜到”。当前并没有完成开放检索闭环。
- 不要把空时间线解释成“传播链不存在”。只能说“当前没有足够材料建立时间线”。

## 6. 现场口播主线

如果你想整场都用同一套句式，最稳的是这 5 句：

1. “我们做的不是普通摘要器，而是一个较真的新闻核查工作台。”
2. “系统会尽量把一条消息拆成几个可验证说法，而不是只给一段自由文本总结。”
3. “当证据比较完整时，我们展示结构化结果和关键传播节点。”
4. “当证据冲突或不足时，我们会显式停在 partial 或 safe，而不是硬给确定性结论。”
5. “所以这次演示重点不是证明系统已经无所不能，而是证明它已经具备可运行、可解释、边界清楚的 V1。”

## 7. 如果被追问，建议怎么答

**Q：你们现在已经能查任意新闻了吗？**

建议回答：

> 还不能这么说。当前最稳定的是三条 demo case，以及规则链路和本地 payload 对齐下的演示闭环。开放场景下真正困难的部分还是 URL 正文抽取、真实检索和传播链还原。

**Q：现在的 verdict 是真实证据驱动的吗？**

建议回答：

> 还不是完全真实检索驱动。当前 provider 主要增强事件理解和 claim 抽取，verdict、evidence、timeline 仍然以规则链路和场景库为主，所以我们在 README 和演示里都会主动把边界说清楚。

**Q：如果后端挂了，为什么页面还可以演示？**

建议回答：

> 当前前端是“真实 analyze 优先、稳定 demo 可兜底”。演示时如果后端离线，预设 demo case 会回退到本地 payload，这样至少三档模式都能稳定演出来，但这不等于真实链路已经全部完成。

## 8. 演示前准备与当前缺口

当前建议的演示前准备顺序：

1. 先按顶层 `README.md` 跑起前后端。
2. 按本文顺序过一遍三条 demo case。
3. 确认现场是否需要联网、是否需要真实 provider。
4. 如果只是求稳，优先演示三条稳定 case，不临场扩展到随机开放输入。

当前还缺一份独立的 `F7` smoke checklist 文档。
也就是说，本文已经能支撑“怎么讲”，但“演示前逐项验收”的正式清单仍应由 `Cluster-F / F7` 补齐。

## 9. 一句话结论

截至 2026-03-13，这个项目已经足够支撑一场 5 到 10 分钟的可信演示；最稳的讲法不是夸它“什么都做完了”，而是把 `complete / partial / safe_mode` 三段讲成一个有边界感的产品闭环。
