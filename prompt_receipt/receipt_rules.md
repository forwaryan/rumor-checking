# AI Prompt Logging Rules (多线程版提示词记录规范)

你是一个严谨的 AI 研发助手。除了执行我下达的代码或文本任务外，你还需要协助我维护一个名为 `prompt-history.md` 的提示词日志文件。

这份规则默认服务于“多个 Codex 窗口 / 多个线程并行工作”的场景。你的目标不只是记录 Prompt，还要帮助我区分不同窗口在做什么、依赖了什么上下文，以及后续应该如何交接。

## 1. 触发条件
- 并非每次对话都需要记录。
- 当我的指令以 `[Log]` 开头，或者我明确提出“记录本次 Prompt”、“更新日志”时，你必须在完成主任务后，生成对 `prompt-history.md` 文件的修改建议（Diff）。
- 如果我明确提到“这是某个分线程 / 某个窗口 / 某个子任务”，即使没有补充更多线程信息，你也要按多线程模式记录。

### 1.1 推荐触发关键词
- 默认触发关键词为 `[Log]`。
- 支持“手动指定线程”和“自动识别线程”两种写法：
  - `[Log]`：仅触发记录，由 AI 根据当前任务自动判断线程标识
  - `[Log][T-main]`：触发记录，并强制把本次任务归入 `T-main`
  - `[Log][T-impl]`：触发记录，并强制把本次任务归入 `T-impl`
  - `[Log][T-impl-ui]`：触发记录，并归入更细的子线程
- 方括号顺序默认采用 `[Log][线程标识]`，线程标识必须放在第二个方括号中，便于稳定识别。
- 如果同时出现 `[Log]` 和线程标识，优先以你手动指定的线程为准，不再自动推断。
- 如果只写了 `[Log]` 而没有线程标识，则按“自动识别线程”规则处理。

### 1.2 自动识别线程规则
- 当未显式标注线程时，你必须根据当前任务内容自动补全线程标识。
- 默认优先级如下：
  - 汇总、决策、协调、规划：归入 `T-main`
  - 搜索资料、竞品分析、信息收集：归入 `T-research`
  - 写代码、改文件、修复问题：归入 `T-impl`
  - 写文档、整理说明、输出汇报材料：归入 `T-doc`
  - 跑测试、做验证、做验收检查：归入 `T-test`
- 如果任务同时跨越多个方向，优先归入“当前窗口正在执行的主动作”，不要为了面面俱到把线程标识写得过于复杂。

### 1.3 兼容表达
- 除了 `[Log][T-xxx]` 形式外，如果我明确写出 `线程=T-xxx`、`thread=T-xxx`、`属于 T-xxx`，也视为手动指定线程。
- 但在没有额外说明时，你应优先推荐并遵循 `[Log][T-xxx]` 这种固定格式，避免歧义。

## 2. 多线程工作原则
- 把每个 Codex 窗口视为一个独立线程，默认**不能假设**它已经继承了其他窗口的上下文。
- 当任务涉及多个线程时，记录里必须体现“这个窗口只负责什么，不负责什么”。
- 每条记录都要尽量回答 3 个问题：
  - 这个线程当前在做什么？
  - 它依赖了哪些已知上下文或输入？
  - 下一步应该交给哪个线程或落到哪个文件？
- 如果我没有提供线程名，你应根据任务自动补一个简洁标识，优先使用以下约定：
  - `T-main`：主控线程，负责汇总、决策、整合
  - `T-research`：调研线程，负责信息收集、竞品分析、资料整理
  - `T-impl`：实现线程，负责编码、改文件、修 Bug
  - `T-doc`：文档线程，负责总结、说明、汇报材料
  - `T-test`：验证线程，负责测试、检查、验收
- 如果任务明显属于某个更细的子问题，也可以使用更具体的名称，例如 `T-impl-ui`、`T-research-claims`。

## 3. 记录格式规范
你必须将本次对话的核心内容，按照以下 Markdown 格式追加到 `prompt-history.md` 文件的末尾：

### 📅 日期时间
> **🧵 线程标识**: [例如 `T-main`、`T-impl-ui`]
> **🏷️ 窗口职责**: [例如 主控 / 调研 / 实现 / 文档 / 测试]
> **🔗 上下文来源**: [本线程依赖的上游任务、已有文件、已有结论；如果没有则写 `当前对话`]
> **💡 原始指令摘要**: [用一两句话精炼我刚才发给你的核心指令内容]

- **🎯 本线程目标 (Context & Goal)**: [说明这个窗口当前要解决的具体问题，而不是笼统复述整个项目目标]
- **🧩 已知约束 (Known Context)**: [记录本线程已知的边界、假设、依赖、未覆盖范围]
- **⚙️ AI 采用的策略 (AI Approach)**: [简述你如何理解并拆解这个指令，以及如何避免和其他线程重复劳动或串线]
- **📦 产出与落点 (Artifacts)**: [记录本线程输出到了哪里，例如某个文件、某个目录、某份结论；没有则写 `待产出`]
- **➡️ 交接建议 (Next Handoff)**: [明确下一步建议谁接手、接手线程该做什么；如果本线程已闭环则写 `当前线程已闭环`]
- **⭐ 效果评估**: [留空，输出 `[待填写]` 即可，留给我后续复盘复写]

---

## 4. 单线程兼容规则
- 即使当前只有一个窗口，也优先按上述格式记录。
- 单线程场景下，默认使用：
  - `线程标识`：`T-main`
  - `窗口职责`：`主控`
  - `上下文来源`：`当前对话`
- 这样后续切成多线程时，历史日志也能直接复用，不需要二次重写。

## 5. 执行要求
- 如果项目根目录下没有 `prompt-history.md` 文件，请先创建它。
- 记录内容必须保持客观、精炼，避免写成流水账。
- **强制使用中文记录**。
- 当同一轮任务在多个窗口并行推进时，优先突出线程差异，不要让不同窗口的记录看起来像重复改写。
- 当主任务已经完成，再补日志；不要为了记录而打断主任务执行。

## 6. 使用示例
- `[Log] 帮我总结当前仓库里的原型文档并沉淀到 requirements/`
  - 含义：触发记录，线程由 AI 自动判断，通常会归入 `T-doc` 或 `T-main`
- `[Log][T-research] 帮我调研腾讯较真AI和类似产品`
  - 含义：触发记录，并强制归入调研线程
- `[Log][T-impl-ui] 帮我把首页信息卡片重构成双栏布局`
  - 含义：触发记录，并强制归入一个更细的实现子线程
- `[Log] 这是一个分线程，主要负责测试接口联通性`
  - 含义：触发记录，AI 会根据任务内容自动归入 `T-test`
