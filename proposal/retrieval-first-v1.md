# Retrieval-First V1 方案评估

## 1. 背景

当前仓库已经不是一个“新闻核查算法原型”，而是一个“可运行、可演示、可回退、可标注来源”的前后端工作台。

这导致代码复杂度同时来自两部分：

- 核查主链路本身
- 为了演示、验收、mock、fallback、provenance、前端展示而增加的工程层

如果当前目标是尽快做出一个真正有价值的 V1，那么“先把核查主链路收缩清楚”比继续扩展工作台更重要。

---

## 2. 用户提出的思路

用户提出的核心思路可以概括为：

1. 用户输入新闻、问题或文本
2. 通过大模型和 prompt 抽取关键信息
3. 根据这些关键信息去做 web search，找到相关公开来源
4. 同时也可以让大模型基于这些信息做问答
5. 综合多个结果，做最终决策和展示

这是一个明显的 **retrieval-first** 思路。

它和当前仓库的主要区别不是“是否使用模型”，而是：

- 当前仓库更像一个多模块工作台
- 这个思路更像一个以检索为核心的事实核查器

---

## 3. 整体判断

### 3.1 这个方向是对的

这个思路比当前实现更接近一个合理的早期产品形态，原因是：

- 主链路更短，更容易跑通
- 更贴近“较真新闻源”的真实工作方式
- 研发重点会聚焦到真正影响质量的部分
- 更适合早期验证，而不是过早搭建配套系统

### 3.2 但原始表述还不够稳

这个思路如果直接按字面落地，容易变成：

- 关键词抽得不准
- 搜索结果噪声很大
- 大模型混入自己的常识和幻觉
- 最后“从多个解里选一个最可能的”，形成伪结论

所以这个思路是正确方向，但需要做几处关键修正。

---

## 4. 这个思路的优点

### 4.1 比当前架构更聚焦

当前项目把大量精力放在这些事情上：

- mock/live/replay/fallback 路径
- provenance 标签体系
- timeline 生成
- 多种演示模式
- 前后端协议兼容与容错

而 retrieval-first 方案会把重点拉回到最核心的 3 件事：

- 用户到底在问什么
- 外部公开来源到底说了什么
- 是否足以支撑结论

### 4.2 更适合做 V1 验证

对一个事实核查产品来说，最先要验证的不是：

- 页面是否足够完整
- provenance 是否足够漂亮
- replay 是否存在

最先要验证的是：

- 给用户一段文本后，系统能不能找到相关证据
- 找到的证据是否真的能帮助判断
- 结果是否比“直接问大模型”更可靠

### 4.3 更容易调试

如果走 retrieval-first，系统问题一般集中在 3 个点：

- query 生成不对
- 搜索结果质量差
- 结论归纳逻辑不稳

这比当前多模块分散式架构更容易定位问题。

---

## 5. 这个思路的主要问题

### 5.1 不能只抽“关键词”

只抽关键词不够，因为关键词会丢掉 claim 的方向性和约束条件。

例子：

- “某公司裁员 40%”
- “某公司回应并未裁员 40%”

这两句话的关键词高度重合，但结论方向完全相反。

更稳的抽取对象应该是：

- 实体：谁
- 动作：发生了什么
- 约束：时间、地点、数量
- 待验证 claim：究竟哪句话在被核查

结论：**关键词可以保留，但不能作为唯一中间表示。**

### 5.2 不能默认“选最可能的一个答案”

这是原始思路里最大的问题。

事实核查系统不能把“多个结果里最像真的那个”当作最终答案，因为很多新闻场景天然存在：

- 证据不足
- 多源冲突
- 旧闻翻炒
- 同名实体误召回
- 权威来源尚未出现

所以最终输出至少应支持四类：

- `supported`
- `refuted`
- `insufficient`
- `conflicting`

也就是说，系统必须允许“不给确定结论”。

### 5.3 第三步不能直接“问 AI”

如果第 3 步只是“把关键词再问给 AI”，会有两个严重问题：

- AI 会混入自己参数里的旧知识
- AI 会在没有证据时补出一个看起来合理的答案

所以更稳的做法不是“再次自由问答”，而是：

- 把检索结果作为证据输入
- 要求模型严格基于证据回答
- 明确给出引用、支持/反驳关系和不确定性

也就是说，第 3 步应当是 **evidence-grounded reasoning**，不是自由聊天问答。

### 5.4 web search 结果本身会很脏

公开搜索常见问题包括：

- 聚合站转载
- 标题党
- 不同事件的同名主体
- 旧闻被重新传播
- 自媒体与权威来源混杂

所以 web search 之后至少还需要一个轻量治理层：

- 去重
- 来源分级
- 时间排序
- 与 claim 的相关性评分

不做这一步，后面的 LLM 只是在消费噪声。

---

## 6. 和当前项目的不同点

### 6.1 当前项目更偏“工作台思路”

当前项目的关注点是：

- 有前后端
- 有 demo 路径
- 有 fallback
- 有 provenance
- 有 mock/live/replay 区分
- 有 timeline 和 report 展示

这是一种“可交付工作台”的思路。

优点是：

- 可演示
- 可验收
- 可联调

缺点是：

- 容易把工程复杂度提前做大
- 研发重心从“核查质量”转移到“系统编排”

### 6.2 用户思路更偏“核查器思路”

用户的思路本质上是：

- 先理解输入
- 再搜索
- 再让模型基于证据判断
- 最后输出结果

这是一种“检索驱动核查器”的思路。

优点是：

- 更接近真实核查动作
- 更适合快速验证质量
- 更适合早期迭代

缺点是：

- 如果不加约束，容易把结果做成“AI 猜测器”

### 6.3 两者最核心的差异

最核心的差异不是“模块多少”，而是 **系统中心到底是什么**：

- 当前项目的中心是“交付工作台”
- 用户思路的中心是“证据检索与判定”

---

## 7. 建议修正版

相比原始 4 步，建议改成下面这版。

### 7.1 修正版流程

1. 用户输入文本、URL 或问题
2. LLM 抽取 `claims + entities + constraints`
3. 为每条 claim 生成 2 到 4 个检索 query
4. 进行 web search
5. 对搜索结果做去重、分级、排序、筛选
6. LLM 严格基于证据输出结论
7. 前端展示结论、证据、风险和引用来源

### 7.2 推荐输出类型

最终不是输出“一个最可能答案”，而是输出：

- 结论：`supported / refuted / insufficient / conflicting`
- 核心 claim
- 关键证据列表
- 为什么得出这个结论
- 当前还缺什么证据

### 7.3 流程图

```mermaid
flowchart LR
    A["用户输入"] --> B["Claim/实体/约束抽取"]
    B --> C["Query 生成"]
    C --> D["Web Search"]
    D --> E["结果去重/分级/排序"]
    E --> F["基于证据的 LLM 判断"]
    F --> G["输出结论 + 证据 + 风险"]
```

---

## 8. 最小可行架构

如果按这个方向做一个更合理的 V1，建议只保留 5 个模块。

### 8.1 `interpret_service`

职责：

- 识别输入类型
- 抽取 claim、实体、时间、地点、数量
- 必要时改写用户问题

### 8.2 `search_service`

职责：

- 根据 claim 生成查询
- 调用 web search
- 对结果做去重、相关性评分、来源分级

### 8.3 `judge_service`

职责：

- 将 claim 和证据打包给 LLM
- 要求其严格基于证据做结论
- 输出支持、反驳、证据不足、冲突

### 8.4 `report_service`

职责：

- 把判定结果转成稳定结构
- 统一输出结论、证据、风险提示

### 8.5 `frontend`

职责：

- 输入区
- 结论区
- 证据列表
- 风险提示

前端不必在 V1 就承担完整的 provenance 状态机、replay 逻辑和复杂 demo 壳。

---

## 9. 这个方向适合当前阶段吗

适合。

原因不是它“更先进”，而是它更符合当前阶段真正需要回答的问题：

- 系统是否真的能找到相关证据
- 系统是否真的能基于证据做出更稳的判断
- 它是否比直接问通用大模型更可靠

如果这三个问题还没回答清楚，就继续在工作台层面加功能，收益会越来越低。

---

## 10. 最终结论

### 10.1 对这个思路的评价

这个思路整体是正确的，而且比当前项目更适合做真实可用的 V1。

### 10.2 必须修正的点

必须修正这三点：

- 不要只抽关键词，要抽 claim 和约束
- 不要默认选“最可能的答案”
- 不要把第 3 步做成自由问答，必须做成基于证据的判断

### 10.3 与当前项目的关系

当前项目不是完全错误，而是工程重心偏向了“工作台化”。

如果后续目标是：

- 尽快验证核查质量
- 降低复杂度
- 缩短迭代路径

那么更合理的方向是：

- 用 retrieval-first 方案重构主链路
- 把 mock/replay/provenance/timeline 降级为次要能力
- 先把“证据驱动判断”做扎实

---

## 11. 后续建议

下一步可以继续做两件事中的一个：

### 方案 A：先写重构蓝图

输出内容：

- 现有模块与新模块的映射
- 哪些文件保留
- 哪些文件合并
- 哪些文件删除或降级

适合先统一方向再动代码。

### 方案 B：直接开始精简实现

优先顺序：

1. 收缩后端主链路
2. 简化返回结构
3. 收缩前端展示
4. 把 mock/replay 从主运行路径剥离

适合已经确定要按这个方向推进。
