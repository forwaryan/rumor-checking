# AI Prompt 资产与评测规则

你是一个强调可解释性和工程可验证性的 AI 研发助手。本文件用于解决一个具体问题：

`prompt-history.md` 适合记录“做了什么任务”，但不足以证明“Prompt 设计是否精巧、异常处理是否充分、AI 工作流是否真的可控”。

因此，本文件定义一套比普通 Prompt 日志更重的“Prompt 资产”规则。目标不是多写文档，而是把 Prompt 相关能力沉淀成面试官可看、可追问、可验证的资产。

## 1. 与 `prompt-history.md` 的分工

- `prompt-history.md`
  - 用途：记录任务时间线、线程、上下文、任务目标和 AI 处理策略
  - 特点：轻量、连续、偏过程留痕
- Prompt 资产
  - 用途：证明 Prompt 设计的版本演进、结构约束、异常处理和评测方式
  - 特点：结构化、可复查、可支撑评分项中的“AI 原生思维与 Prompt”

换句话说：

- `prompt-history.md` 解决“我们做过什么”
- Prompt 资产解决“我们为什么这样设计，以及它是否真的有效”

## 2. 何时必须进入 Prompt 资产模式

当任务涉及以下任一情况时，不能只记录普通 Prompt 日志，还必须补 Prompt 资产：

- 设计或修改关键 Prompt
- 调整多阶段 AI pipeline
- 定义或修改结构化输出 schema
- 处理 hallucination / 证据不足 / 冲突证据等风险
- 处理长文本、上下文超限、分块摘要等问题
- 做 Prompt 调优、回归验证或 case 评测
- 需要向评分项证明“AI 不是乱生成，而是被约束地工作”

## 3. 必须沉淀的 6 类 Prompt 资产

### 3.1 关键 Prompt 版本

至少记录：

- Prompt 名称
- 版本号，例如 `v1`、`v2`
- 所属阶段，例如 `claim_extraction`、`evidence_selection`
- 目标任务
- 这版相对上一版改了什么
- 为什么要改

要求：

- 不能只说“优化了 Prompt”
- 必须写出版本之间的变化原因

### 3.2 Prompt 失败问题与修复记录

至少记录：

- 失败案例
- 失败表现
- 判断的根因
- 对应改动
- 改动后效果

要求：

- 要能回答“这个 Prompt 是为了解决什么失败问题”
- 没有失败记录的 Prompt，通常很难证明它经过了有效调优

### 3.3 输出 schema

至少记录：

- 输出字段名
- 字段含义
- 字段类型
- 必填/选填
- 哪些字段决定前端渲染或下游逻辑

要求：

- 不能只有自然语言描述
- 应尽量给出 JSON 示例、表格 schema 或显式字段列表

### 3.4 幻觉防控策略

至少记录：

- 哪些场景禁止模型强行下结论
- 什么时候必须输出 `insufficient`、`mixed`、`conflicting`
- 模型结论和证据之间如何绑定
- 哪些字段必须来自检索证据，不能自由脑补

要求：

- 要能回答“为什么这个系统不会随口判断”
- 不能只写“减少幻觉”，必须写触发条件和约束方式

### 3.5 上下文超限处理策略

至少记录：

- 什么输入会导致超长
- 如何分块
- 如何摘要
- 如何保留关键事实和时间信息
- 分块后的合并逻辑

要求：

- 要能回答“新闻太长时系统怎么处理”
- 不能只写“分段处理”，要写分段依据和合并原则

### 3.6 评测方式

至少记录：

- 测试 case 名称或编号
- case 类型
- 预期输出
- 观察指标
- 是否通过

要求：

- 至少覆盖：正常 case、证据不足 case、冲突证据 case、长文本 case
- 要能回答“你怎么知道 Prompt 比之前更好了”

## 4. 最小可交付标准

如果当前项目时间紧张，不要求一开始做完整 Prompt 平台，但至少要满足以下最低标准：

1. 有 3 到 5 个关键 Prompt 的版本记录
2. 有 3 条以上失败问题与修复记录
3. 有一份核心输出 schema
4. 有一份幻觉防控策略说明
5. 有一份上下文超限处理说明
6. 有一张小型 eval 表，覆盖至少 4 类 case

如果连这 6 项都没有，就很难把 “AI 原生思维与 Prompt” 讲成一个可信的高分项。

## 5. 当前项目最值得优先沉淀的 Prompt 资产

结合当前“较真新闻观察员”项目，优先级建议如下：

1. `claim_extraction`
   - 负责把新闻文本拆成可核查 claim
2. `claim_classification`
   - 负责区分 fact / opinion / prediction / unverifiable
3. `evidence_selection`
   - 负责从检索结果里挑选高相关证据
4. `verdict_generation`
   - 负责基于证据输出 `supported / refuted / insufficient / conflicting`
5. `event_summary_or_timeline`
   - 负责整合传播链关键节点与事件摘要

如果时间不够，至少先把前 4 个模块的 Prompt 资产做出来。

## 6. 与工作流的协同要求

- 轻日志仍然走 `workflows/prompt_logging_rules.md`
- 一旦进入 Prompt 设计、调优、评测任务，必须同时遵循本文件
- 模板参考统一放在 `requirements/guides/04_prompt_inventory.md`

## 7. 输出要求

当我让你“补 Prompt 资产”“把 Prompt 设计做成可展示材料”“增强 AI 原生思维评分支撑”时，你应优先输出：

- 版本记录
- 失败修复记录
- schema
- 幻觉防控规则
- 上下文超限策略
- eval case 表

而不是只给一段泛泛而谈的 Prompt 设计说明。
