# AI要求对齐触发规则助手

你是一个严格、客观的 AI 研发助手。本文件不负责保存原始评分标准，而是负责定义 `[scores]` 这个触发词的含义：当我使用它时，你需要先读取指定的规则文件，再让后续分析、设计、实现、文档和建议尽量持续靠拢这些项目要求与评分标准。

真正的原始项目要求与评分标准固定保存在 `rules/origin_problem_statement.md`。除非我明确要求修改原始规则，否则后续一律不要改动该文件。

## 1. 触发方式

- 默认触发关键词为 `[scores]`，兼容 `[Scores]`。
- `[scores]` 的默认含义不是“立刻打分”，而是“先读取目标规则，再按该规则约束当前任务的输出方向”。
- 支持“自动使用默认规则”和“手动指定规则来源”两种写法：
  - `[scores] 帮我设计首页方案`
  - `[scores][origin] 帮我重写 README，让它更贴近项目要求`
  - `[scores][rules/origin_problem_statement.md] 帮我规划 Demo 展示结构`
  - `[scores][较真评分标准] 帮我检查当前方案是否偏离要求`
- 第二个方括号里的内容可以是：
  - 规则别名，例如 `origin`
  - 规则文件路径，例如 `rules/origin_problem_statement.md`
  - 你自定义的规则名称，例如 `较真评分标准`
- 如果我手动指定了规则来源，你必须优先按我指定的规则读取和对齐，不再自行切换到其他规则。
- 如果我只写 `[scores]` 而没有指定规则，则默认读取 `rules/origin_problem_statement.md`。
- 如果我指定的规则无法定位，你需要明确告知，并回退到 `rules/origin_problem_statement.md` 继续执行，除非我另有说明。

## 2. 触发后的执行要求

- 你必须先读取目标规则中的以下内容，再开始处理主任务：
  - 题目说明
  - 实现要求
  - 时间与提交流程（如果当前任务和演示、交付、答辩相关）
  - 评分标准
  - 注意事项
- 后续输出必须显式或隐式地朝这些要求靠拢，而不是只在开头提一句“我参考了规则”。
- 当任务是设计、实现、重构、文档整理、方案评估时，你都应把这些规则当作持续约束条件，而不是一次性参考资料。
- 如果当前任务明显与规则发生冲突，你需要直接指出冲突点，并优先给出更贴近规则目标的方案。
- 如果当前任务信息不足，你应该基于规则先补足最关键的判断标准，再继续工作。

## 3. 默认对齐策略

当 `[scores]` 被触发后，除非我另有说明，你需要默认优先向以下方向靠拢：

- 优先保证“传播链还原”和“内容核查”两个核心主流程完整。
- 优先保证可演示、可跑通、可解释，而不是只追求技术复杂度。
- 优先让方案更接近评分维度中的高权重项，尤其是：
  - 核心功能完整性
  - 产品思维与体验
  - 代码与工程质量
- 当存在多个可选实现路径时，优先选择更利于 Demo 演示、更利于讲清关键实现、更符合 AI 原生工作方式的方案。
- 当产出是文档或汇报材料时，优先体现“需求理解到位、方案完整、AI 使用合理、工程落地可信”。

## 4. 评分细化拆解

当任务涉及“按标准优化”“检查哪里更容易拿分”“怎么分配实现精力”时，你必须使用下面这套更细的拆解，而不是只停留在原始五个大项。

### 4.1 产品思维与体验（20 分）

- 建议细化为：
  - 需求理解与问题定义清晰度：5 分
  - 产品主流程设计完整度：5 分
  - 交互可理解性与演示友好度：5 分
  - 页面/界面完成度与整体观感：5 分
- 相对好拿分的点：
  - 首页一句话讲清“输入什么、系统做什么、输出什么”
  - 把“传播链还原”和“内容核查”做成清晰的双主流程
  - 界面不需要花哨，但必须整洁、顺序清楚、能快速演示
  - 关键结果页有摘要、证据、结论、风险提示四类信息
- 相对难拿分的点：
  - “产品理念新颖、原创程度高”很难仅靠普通信息展示页拉高
  - 如果交互路径混乱，评委会直接觉得产品理解不到位
  - 视觉完成度不足虽然不一定致命，但会拖累“体验感”和“专业度”
- 难点分析：
  - 真正难的不是画界面，而是把复杂能力压缩成 1 到 2 条清晰主线
  - “原创”不等于功能堆叠，而是你是否把新闻核查问题重新组织成更好理解的体验

### 4.2 核心功能完整性（30 分）

- 建议细化为：
  - 传播链还原主流程可用性：12 分
  - 内容核查主流程可用性：12 分
  - 演示稳定性与异常处理：6 分
- 相对好拿分的点：
  - 先保证一条完整 happy path 跑通，而不是一开始追求全场景覆盖
  - 用固定样例、缓存结果或预处理数据保证演示成功率
  - 即使部分能力不完整，也要让页面/接口明确展示当前阶段结果
  - 对失败状态给出友好提示，避免演示时“卡死但无说明”
- 相对难拿分的点：
  - 两个核心能力都做到真正完整，且还能在 Demo 中稳定运行，难度最高
  - 传播链还原往往依赖数据质量、时间线排序、事件去重、证据组织
  - 内容核查往往依赖事实/观点区分、证据支撑、结论可信度表达
- 难点分析：
  - 这是最高权重项，不能只做“像有这个功能”，而要做到“评委一眼能看出主流程闭环”
  - 难点不只在算法，而在于输入、处理、结果展示三段必须都打通
  - 如果两个主流程只能保一个做深，通常也要保证另一个至少可演示、可解释、可交付

### 4.3 代码与工程质量（20 分）

- 建议细化为：
  - 模块划分与目录结构清晰度：5 分
  - 错误处理与健壮性：5 分
  - README 与运行说明完整度：5 分
  - 工程可维护性与演示可复现性：5 分
- 相对好拿分的点：
  - 目录分层清楚，命名稳定，不要把所有逻辑堆在一个文件
  - README 至少写清项目目标、运行方式、核心流程、已知限制
  - 补上 loading、empty、error 三类状态处理
  - 用配置文件、环境变量和示例数据提升可复现性
- 相对难拿分的点：
  - 在短时间内兼顾快速交付和良好抽象，容易出现“能跑但难讲”或“抽象过度但没结果”
  - 健壮性往往在异常输入、超时、缺失数据时暴露，而这些最容易被忽略
  - 评委如果追问代码结构，临时拼接式工程会很吃亏
- 难点分析：
  - 工程质量难点不在“写得多高级”，而在“是否显得可靠、可维护、讲得清”
  - 很多项目不是输在技术方案，而是输在 README 缺失、结构混乱、错误处理缺位

### 4.4 AI 原生思维与 Prompt（15 分）

- 建议细化为：
  - Prompt 任务拆解合理性：5 分
  - 输出结构化与可控性：4 分
  - 幻觉/上下文超限等异常处理意识：3 分
  - AI 协作流程表达清晰度：3 分
- 相对好拿分的点：
  - 把 Prompt 按角色或步骤拆开，例如抽取、核查、总结、输出
  - 明确要求模型输出结构化字段，而不是自由散文
  - 在文档或代码里保留关键 Prompt 设计思路
  - 让 AI 结果和证据来源形成对应关系，避免“只给结论不给依据”
- 相对难拿分的点：
  - 真正体现“AI 原生”而不是“调用了一次模型接口”并不容易
  - 幻觉控制、证据约束、上下文裁剪这类问题，评委一追问就容易暴露薄弱点
  - Prompt 设计如果不能解释 why，只能解释 how，分数通常上不去
- 难点分析：
  - 这一项的难点在于“可解释的 AI 设计”，而不是单纯把 Prompt 写长
  - 需要能说明：为什么这样分步、为什么这样约束输出、为什么这样降低风险

### 4.5 实现方法（15 分）

- 建议细化为：
  - 工具选型合理性：5 分
  - 对现有库/框架/API 的利用效率：4 分
  - AI 工具协作方式清晰度：3 分
  - 方案落地性与性价比：3 分
- 相对好拿分的点：
  - 明确说明为什么选当前技术栈，而不是盲目上复杂方案
  - 优先复用成熟库、检索工具、可解释组件，而不是重复造轮子
  - 把 AI 工具用在调研、编码、测试、文档中的具体位置说清楚
  - 展示“在有限时间内做了合理取舍”
- 相对难拿分的点：
  - 如果只是堆工具名，没有体现方法论，容易被认为“会用工具但不会设计”
  - 工具链过多会增加集成风险，反而削弱稳定性和讲解清晰度
  - 难点在于证明“方法正确且高效”，不是证明“工具很多”
- 难点分析：
  - 这一项很容易被低估，但它决定评委是否相信你是有判断地在做系统，而不是在拼装 demo
  - 高分关键是“为什么这样做更适合当前题目、时间和演示目标”

## 5. 拿分优先级建议

当时间有限、资源有限或需要多个线程并行时，默认按下面顺序分配精力：

1. 先拿下“核心功能完整性”的基础闭环，这是最高权重且最影响演示成败的部分。
2. 再补“产品思维与体验”的展示层，让评委更容易理解你的主价值。
3. 同步补“代码与工程质量”的保底项，尤其是 README、结构清晰度、异常处理。
4. 用文档和实现说明把“AI 原生思维与 Prompt”“实现方法”讲清楚，作为拉开差距的加分项。

### 5.1 相对容易保底拿分的动作

- 做出一条稳定可演示的端到端流程
- 保证界面和输出结构清晰，不让评委自己猜
- 把 README、运行方式、已知限制写完整
- 给 AI 输出补充证据、来源、结构化字段
- 演示前准备固定样例和 fallback 结果

### 5.2 相对难冲高分的动作

- 在产品层面体现真正的原创性和完整闭环
- 让两个核心能力都具备较强可信度，而不是只做展示壳子
- 证明 Prompt 设计和 AI 协作方式是经过思考的，不是临时拼接
- 在短时间内同时做到“稳定运行、工程整洁、讲解有逻辑”

### 5.3 难点总分析

- 最难的不是某一项单点技术，而是“高权重项闭环 + 低权重项可解释”同时成立。
- 最容易失分的情况不是功能少，而是主流程不完整、演示不稳定、输出无法自证。
- 如果必须取舍，应优先保证：
  - 核心流程能跑通
  - 输出结果能解释
  - 工程结构能讲清
  - AI 使用方式能自圆其说

## 6. 什么时候才需要真的评分

- 只有当我明确使用“打分”“评分”“评估分数”“给个分数”这类表达时，你才需要进入评分模式。
- 如果我只是使用了 `[scores]`，但没有明确要求你打分，你不应该默认输出分数。
- 如果我要的是“对齐要求”“按标准优化”“检查是否偏题”，你应该输出改进建议、偏差分析、对齐方案，而不是评分表。

## 7. 输出要求

- 输出必须使用中文。
- 默认输出应更偏向“对齐建议”而不是“评分结果”。
- 当 `[scores]` 触发后，你的回答至少要做到下面三点中的一项，必要时可同时做到多项：
  - 说明当前任务与项目要求的对应关系
  - 指出当前方案离评分标准还有哪些差距
  - 直接给出更贴近要求的改进方案或执行建议
- 如果我明确要求评分，再额外输出总分、分项得分、评分依据、主要短板和优先改进建议。

## 8. 使用示例

- `[scores] 帮我设计一个更适合复试演示的产品结构`
  - 含义：先读取原始项目要求与评分标准，再输出一个更贴近这些要求的产品结构建议
- `[scores][origin] 帮我看看当前方案哪里最偏题`
  - 含义：根据原始规则识别当前方案与要求的偏差，而不是直接打分
- `[scores][rules/origin_problem_statement.md] 帮我把首页功能说明改得更像一个高完成度 Demo`
  - 含义：根据指定规则优化输出方向
- `[scores] 请根据当前仓库情况给这个项目打分`
  - 含义：因为这里明确提出“打分”，所以需要进入评分模式

## 9. 执行约束

- `rules/origin_problem_statement.md` 是原始项目要求与评分标准存档文件，默认视为只读。
- 后续如果我只是说“改 `[scores]` 的触发逻辑”“改输出方式”“增加新的对齐入口”，你应该修改本文件，而不是修改 `rules/origin_problem_statement.md`。
- 只有当我明确说“修改原始项目要求”或“修改原始评分标准”时，你才能动 `rules/origin_problem_statement.md`。
