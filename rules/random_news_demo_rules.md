# 随机新闻 Demo 应对规则

你是一个强调稳定性、边界感和可解释性的 AI 研发助手。本文件用于解决一个现实问题：

在复试 Demo 或临场演示时，面试官可能不会使用预设 case，而是随机给出一条新闻、一个 URL，或一句“XXX 是真的吗”。

这类场景下，系统的目标不是“承诺一定查清真相”，而是：

1. 稳定接住随机输入
2. 在有限时间内给出结构化结果
3. 证据不足时明确承认边界
4. 不因随机 case 翻车而显得系统不可靠

## 1. 核心原则

- 随机新闻场景下，优先保证“有边界地返回结果”，而不是强行返回完整结论。
- 高分关键不是“任何新闻都 100% 判断正确”，而是“任何新闻都能稳定进入合适的处理模式”。
- 系统应优先避免以下失分行为：
  - 没证据却强判真假
  - 明显是观点内容却硬做事实判断
  - 搜索失败后直接空白或报错
  - 传播链不完整却伪装成完整还原

## 2. 随机输入下的默认处理目标

当输入是随机新闻时，默认只承诺做到以下三件事：

1. 把输入标准化成一个待分析事件
2. 给出当前可拿到的证据层级与初步结构化判断
3. 根据证据充足度，选择“完整模式 / 部分模式 / 安全模式”之一

## 3. 输入分流规则

### 3.1 输入类型识别

随机输入默认先分为以下四类：

- `url_news`
  - 标准新闻网页链接
- `url_unknown`
  - 非标准新闻页、聚合页、社交页、截图页或抽取不稳定页面
- `text_news`
  - 用户直接粘贴新闻正文或摘要
- `question_only`
  - 用户只问一句“XXX 是真的吗”，没有给原文和链接

### 3.2 输入优先级

处理优先级应为：

1. `text_news`
2. `url_news`
3. `question_only`
4. `url_unknown`

原因：

- 文本输入最稳定，最适合 claim 抽取
- 标准新闻 URL 次之
- 只有问题时可做分析，但上下文最少
- 非标准 URL 最容易抽取失败

### 3.3 输入失败 fallback

如果 URL 抽取失败：

- 不要直接终止
- 立刻切换为：
  - 页面标题 + 摘要分析，或
  - 请求用户粘贴正文，或
  - 只基于问题本身进入安全模式

## 4. 证据充足度分级

随机新闻场景下，必须先判断证据层级，再决定是否可以下结论。

### 4.1 证据等级

- `A`
  - 有官方通报、原始声明、权威机构或多家高可信来源交叉支持
- `B`
  - 有多家主流媒体或可信来源报道，但缺少一手源
- `C`
  - 只有少量二手报道、自媒体转述或证据链较弱
- `D`
  - 基本搜不到有效证据，或证据高度冲突

### 4.2 证据等级与输出边界

- `A/B`
  - 可以进入完整模式或部分模式
- `C`
  - 只能谨慎进入部分模式，优先输出 `insufficient` 或 `conflicting`
- `D`
  - 必须进入安全模式，禁止输出确定性 verdict

## 5. 三档输出模式

### 5.1 完整模式

适用条件：

- 输入抽取稳定
- 证据等级为 `A` 或较强 `B`
- 能拿到 3 条以上有效证据
- 能抽出 3 到 5 条有效 claim

输出要求：

- 事件概览
- 传播链关键时间线
- claim 核查表
- 综合结论
- 证据来源列表

### 5.2 部分模式

适用条件：

- 输入基本可用
- 证据等级为 `B` 或 `C`
- 部分 claim 能判断，部分不能判断
- 传播链只能还原部分关键节点

输出要求：

- 事件概览
- 简版时间线
- 部分 claim 核查表
- 明确标注哪些部分证据不足
- 综合结论必须保守表达

### 5.3 安全模式

适用条件：

- 输入抽取失败或文本不足
- 证据等级为 `D`
- 当前事件过新，公开证据不足
- 搜索失败或结果高度冲突

输出要求：

- 事件/问题摘要
- 当前可确认的信息
- 当前无法确认的信息
- 建议进一步补充的输入或证据方向
- 显式说明：当前不能给确定性结论

## 6. Verdict 硬边界

随机新闻场景下，必须遵守以下硬规则：

- 没有足够证据时，禁止输出 `supported` 或 `refuted`
- 观点、评论、预测类内容，优先标记为 `opinion` 或 `unverifiable`
- 证据冲突时，优先输出 `conflicting`
- 公开证据不足时，优先输出 `insufficient`
- 系统不能为了“显得聪明”而补齐不存在的证据链

## 7. 传播链硬边界

随机新闻场景下，传播链默认只承诺做到“关键来源时间线”，不承诺完整社交传播图。

### 7.1 可以稳定承诺的内容

- 最早可见来源
- 若干关键跟进报道
- 高峰或转折节点
- 是否出现官方回应 / 辟谣 / 补充说明

### 7.2 不应强行承诺的内容

- 完整转发关系图
- 精确传播路径的强因果判断
- 所有平台上的首发源头

### 7.3 传播链不足时的安全表达

- “当前仅能还原公开新闻来源层的关键节点”
- “以下时间线基于当前可检索公开报道，不代表完整传播全貌”

## 8. Demo 现场应对话术建议

当随机新闻质量一般或证据不足时，答辩时应主动这样讲：

- “这个系统的设计目标不是对所有随机输入强判真假，而是先给出证据驱动、边界明确的分析结果。”
- “当前这条新闻在公开证据层面只能支持部分判断，所以系统进入了部分模式 / 安全模式。”
- “这恰恰体现了产品的较真属性：有证据才判断，没证据就明确保留。”

## 9. 工程实现要求

如果要让这套规则真正可用，至少要实现以下工程机制：

- URL 抽取失败 fallback
- 检索失败 fallback
- 部分结果可返回，而不是全有或全无
- verdict 标签固定为：
  - `supported`
  - `refuted`
  - `insufficient`
  - `conflicting`
- 页面必须显式展示：
  - 当前模式
  - 当前证据层级
  - 当前结论边界

## 10. 随机 case 评测要求

要证明系统能应对随机新闻，不能只测预设 Demo case。

最小评测建议：

1. 选 15 到 20 条陌生新闻
2. 覆盖以下类型：
   - 标准新闻报道
   - 热点争议事件
   - 证据不足新闻
   - 观点评论类内容
   - 长文本新闻
   - URL 抽取不稳定页面
3. 对每条 case 记录：
   - 输入类型
   - 证据等级
   - 输出模式
   - 是否误判
   - 是否能稳定给出边界清晰的结果

## 11. 评分对齐解释

这份规则主要服务以下评分项：

- 核心功能完整性
  - 随机输入下仍能跑通主流程或安全降级
- 产品思维与体验
  - 用户看到的不是报错，而是边界清晰的分析结果
- AI 原生思维与 Prompt
  - 体现不乱判、可解释、可控输出
- 实现方法
  - 体现你知道如何在不确定输入下设计稳定系统

## 12. 最终目标

随机新闻 Demo 的目标不是“永远答对”，而是：

- 永远不乱答
- 尽可能多答
- 答不出时也能讲清为什么

做到这一点，系统就已经具备了较强的答辩稳定性。
