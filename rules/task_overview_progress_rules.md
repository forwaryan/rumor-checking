# 任务总览表进度规则

你是一个强调状态真实、依赖透明和持续回写的 AI 研发助手。本文件用于解决一个具体问题：

实时记录目前完成情况的总表时，某个主 task 往往不是单点完成，而是依赖多个子 task 共同推进。如果总表只写一句“进行中”或“已完成”，读者无法判断这个主 task 到底卡在哪个依赖项上。

因此，当使用 `[task_rules]` 时，必须把主 task 依赖到的子 task 一并展开，并标注每个子 task 的当前进度。

## 1. 触发条件
- 并非每次对话都需要进入总表联动维护。
- 当我的指令明确提到 `[task_rules]`，或者我明确提出“更新任务总览表”“更新总表”“同步全局任务矩阵”“实时记录目前完成情况的总表”“把主 task 依赖的子 task 和进度写进总表”时，你必须按本规则执行。
- 如果我只是让你分析任务拆分、讨论进度判断或设计总表结构，但没有要求回写总表，你可以先输出分析或建议，不要擅自改动总表文件。

## 2. 适用范围

- `tasks/origin-problem-goal-matrix.md`
- 其他承担“实时总表 / 总览表 / 全局任务矩阵”职责的文档

## 3. 核心原则

- 总表中的主 task 状态，必须能被子 task 状态解释。
- 依赖多个子 task 的主 task，不能只保留一句模糊进度。
- 子 task 状态一旦变化，主 task 汇总状态和联动表要同轮同步更新。
- 若文档口径与代码、测试或正式验收记录冲突，以代码、测试和正式验收记录核实后的结果为准。

## 4. 强制规则

1. 主 task 依赖多个已有 task id 时，必须显式列出所有依赖子 task，优先使用正式编号，如 `C11`、`D5`、`F8`。
2. 每个子 task 旁边都要写当前状态或进度，不能只写 task id，不允许把子 task 进度省略掉。
3. 主 task 的汇总状态必须和子 task 状态一致；若关键子 task 仍未完成，主 task 不得直接写成“已完成”。
4. 某个子 task 状态变化时，要在同一轮同步回写主 task 汇总状态和联动表，不能只改其中一处。
5. 联动关系写在总表内维护，不单独挪到归档文件；归档只用于问题登记，不替代实时状态。
6. 若主 task 已形成正式记录，但结论是“未通过最终验收”，总表中仍应写“记录已完成，但结论未通过”，不能误写成能力完成。

## 5. 推荐表头

| 主 task | 完成该 task 需要的子 task | 子 task 当前进度 | 主 task 汇总状态 | 备注 |
| --- | --- | --- | --- | --- |

## 6. 维护要求

- 子 task 列应覆盖“当前真正决定主 task 状态”的依赖项，不写无关任务。
- 若某个子 task 已完成但只完成第一阶段，要写成“第一阶段完成”或同等真实表述，不能省略尾项风险。
- 当一个主 task 需要跨 cluster 子 task 共同完成时，优先在任务总览表内集中维护。
- 若已经存在主任务矩阵，联动表应与主矩阵放在同一文件中维护，不另开旁路状态文件。

## 7. 示例

| 主 task | 完成该 task 需要的子 task | 子 task 当前进度 | 主 task 汇总状态 | 备注 |
| --- | --- | --- | --- | --- |
