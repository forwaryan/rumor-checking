# Scoped Append-Only Task File Update Rules

你是一个在多线程 / 多 Agent 协作环境中维护共享任务文档的 AI 助手。本文件用于解决一个具体问题：

共享 Markdown 任务板、进度文档和研究记录，容易被某个线程为了“整理”“总结”或“精简”而误删旧内容，导致其他线程依赖的上下文、历史判断和执行轨迹丢失。

因此，本文件定义一套可通过关键字触发、带作用域隔离的 Append-Only 更新协议。目标不是让文档变短，而是让文档在多人 / 多线程协作下保持可追溯、可审计、可回放。

## 1. 触发条件

- 当我的指令显式包含 `[task_file_rules]` 时，你必须启用本规则。
- 当启用本规则时，Prompt 中必须包含 `[TARGET_TASK_FILE: <文件路径>]`，表示你被授权更新的目标文件。
- 可选附加 `[TARGET_TASK_SECTION: <标题 / task id / 区块标识>]`，用于把你的可写范围进一步收敛到目标文件中的某个区块。
- 如果我要求你“同步任务文档”“更新任务板”“维护共享 Markdown 任务文件”，且上下文里已经给出明确目标文件，也应按本规则执行。
- 如果缺少 `TARGET_TASK_FILE`，且当前上下文无法唯一确定目标文件，不要猜测，不要擅自修改任何共享任务文档。

## 2. 适用范围

- 所有承担“共享任务状态 / 进度记录 / 研究轨迹 / 多线程协作上下文”职责的 Markdown 文档。
- 不仅限于 `tasks/` 目录；只要某份文档被多个线程共享，就应视为本规则适用对象。
- 本规则优先保护已有内容不被删除；即使目标文件不是传统任务板，只要承载共享上下文，也应遵守 Append-Only 原则。

## 3. 核心原则

- Append-Only：只能追加、标记状态、补充说明、修正表达；不能通过物理删除来“清理”历史。
- Traceable：每一次状态变化、取消决策和补充判断，都要保留可追溯痕迹。
- Scoped：你只修改被明确授权的目标文件 / 目标区块，不越权碰其他共享文档。
- Structure-Preserving：保持原有标题层级、表格、Mermaid、缩进和任务树结构稳定。
- Evidence-Based：任务完成、取消或修正时，必须留下简短依据，而不是只改一个状态符号。

## 4. 强制规则

### 4.1 作用域隔离（Strict Scoping）

- 你只允许更新 `[TARGET_TASK_FILE: ...]` 指定的目标文件。
- 如果同时给出 `[TARGET_TASK_SECTION: ...]`，你只允许更新该区块及其直接相关的新增补充内容。
- 对仓库中的其他共享任务文档、进度文档、计划文档，你只能读取，不能顺手改动。
- 除非用户明确授权，不要把一个文件中的任务迁移、合并或重写到另一个文件。

### 4.2 绝对禁止物理删除（No Deletion）

- 任何情况下，不得删除目标文件中已有的标题、表格行、任务项、注释、记录、结论或历史说明。
- 不得因为“已经完成”“已经过时”“内容重复”“想让文档更简洁”而删除旧内容。
- 不得用“整体重写”“重新整理”“摘要替换原文”的方式覆盖既有任务内容。
- 如果原有表达不准确，也不能直接抹掉后重写，必须保留旧判断痕迹。

### 4.3 任务状态的合法流转（Status Mutation）

- 你可以通过更新 Checkbox 状态推进任务：
  - `[ ] -> [-]` 表示开始进行中
  - `[-] -> [x]` 表示已完成
- 如果某任务原本没有显式状态，但文件结构允许补充状态，可在不破坏原句的前提下追加标准状态标记。
- 当任务被标记为 `[x]` 时，应在同一行追加简短完成证明，推荐格式：
  - `[x] T1.1 修后端测试 (Done: 修复了 xxx_test.py，相关测试通过)`
- 不能把仍依赖未完成子任务的主任务直接写成“已完成”。

### 4.4 错误判断与废弃任务的处理（Strikethrough for Deprecation）

- 如果你发现某个任务方向错误、判断有误、路径已废弃或策略已改变，绝对不准删除原任务。
- 必须使用 Markdown 删除线保留原内容，并在后面追加取消 / 修正原因。
- 推荐格式：
  - `~~[ ] T2.3 强依赖外网兜底~~ (取消：评估后决定改为内部 fallback)`
  - `~~结论：该线索可直接判真~~ (修正：证据不足，暂降级为 insufficient)`
- 如果是局部表述错误，也应优先追加“修正说明”或“更正 Note”，而不是静默改写到看不出历史。

### 4.5 任务的丰富与裂变（Enrichment & Expansion）

- 如果原任务粒度过大，需要拆分子任务，保持原任务项不动，在其下方按层级追加新的子任务。
- 推荐子任务格式：
  - `- [ ] a. ...`
  - `- [-] b. ...`
  - `- [x] c. ... (Done: ...)`
- 如果需要记录新发现、风险、依赖、阻塞点或上下文说明，追加在对应任务块下方，推荐使用引用格式：
  - `> Note: ...`
  - `> Blocker: ...`
  - `> Decision: ...`
- 不要把新增发现挤压成对旧内容的覆盖式改写。

### 4.6 尊重文件结构（Respect Document Structure）

- 严禁破坏原有的 Markdown 层级、任务缩进、表格列含义、Mermaid 代码块或文档布局。
- 如果必须补充表格内容，应在现有列语义内增补，不要随意重排整张表。
- 如果原文件使用的是任务树、日志式记录、章节式规划或矩阵表，你的更新应延续原结构，而不是切换成另一套格式。
- 不要为了“更整洁”而折叠、合并、搬移已有区块。

### 4.7 冲突信息的处理（Correction Over Erasure）

- 当新研究结果与旧记录冲突时，优先采用“追加修正”而不是“抹去旧结论”。
- 你应清楚标明：
  - 哪条旧判断被修正
  - 修正原因是什么
  - 新结论当前是否稳定
- 如果尚不能完全确认，不要把旧内容删除后换成另一个同样可能不稳定的判断。

## 5. 禁止行为

以下行为在本规则下均视为违规：

- 删除旧任务、旧标题、旧注释、旧表格行
- 把多个历史任务压缩成一行“已完成”
- 用摘要替换原任务明细
- 因为任务完成就移除其子任务
- 未经授权修改非目标文件中的共享任务内容
- 为了统一格式而重写整份目标文档
- 发现错误后直接改写原句，让后来者无法看出修正轨迹

## 6. 与其他规则的协同

- 如果同时触发 `[task_rules]`，先遵守本文件的“不删除 + 作用域隔离 + 保留轨迹”，再执行主 task / 子 task 联动展开规则。
- 如果同时触发其他与提交、评分、Prompt 资产相关的规则，本文件只负责“共享任务文档如何安全更新”，不替代其他规则本身的业务要求。
- 本规则是全局文档维护规则，不仅适用于某一份任务板，而是适用于所有线程共享的任务 / 进度 / 研究文档。

## 7. 推荐调用方式

推荐在给线程或子 Agent 的 Prompt 末尾附加如下标签：

```text
[task_file_rules]
[TARGET_TASK_FILE: tasks/multi-agent-execution-board.md]
```

如果只允许它更新某一段，可进一步追加：

```text
[task_file_rules]
[TARGET_TASK_FILE: tasks/multi-agent-execution-board.md]
[TARGET_TASK_SECTION: T2]
```

## 8. 推荐更新示例

### 8.1 推进任务

```md
- [-] T2.1 核查法官身份
  - [x] a. 对齐姓名拼写 (Done: 已核对官方简介与新闻稿的英文拼写)
  - [ ] b. 补充法院任职来源
  > Note: 现有两处来源对中间名写法不一致，先保留待复核。
```

### 8.2 取消错误方向

```md
~~[ ] T2.3 强依赖外网兜底~~ (取消：评估后决定改为内部 fallback，避免演示期网络波动)
```

### 8.3 修正错误判断

```md
~~结论：该传播链已闭环~~ (修正：目前只闭环到二跳节点，三跳节点仍缺证据)
> Decision: 暂不宣称完整闭环，等待补证。
```

## 9. 最低执行标准

当 `[task_file_rules]` 被触发时，你至少必须保证：

1. 不删除目标文件中的既有内容
2. 不越权修改其他共享任务文件
3. 所有完成项都有简短完成证明
4. 所有取消 / 修正项都保留删除线和原因
5. 所有新增内容都以追加方式进入原有结构

如果做不到这 5 点，就不算遵守了本规则。
