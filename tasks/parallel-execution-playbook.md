# Parallel Execution Playbook

更新时间：2026-03-13 21:23（Asia/Shanghai）

## 1. 当前推荐执行模型

当前不再推荐“每个很小的子 task 一个 prompt”的模式。

当前推荐模型是：

- 每个 `cluster-*.md` 文件各自提供一个“窗口执行 Prompt（全局）”
- 一个窗口拿到一个 cluster 文件后，直接按该文件里的全局 prompt 开始工作
- 窗口真正执行前，再根据该文件里“进行中 / 未完成”的子任务，把当前要做的一块拆成更细步骤

这样做的目标是：

- 减少 prompt 数量
- 保持任务边界稳定
- 让窗口可以根据文件现状自主推进，而不是等主控为每个小动作单独写 prompt

## 2. 如何使用

1. 先打开对应的 `tasks/cluster-*.md`
2. 直接复制其中的 `窗口执行 Prompt（全局）`
3. 把这个 prompt 发给执行窗口
4. 执行窗口先阅读当前状态、实现备注和未完成子任务
5. 执行窗口先把本轮目标再拆成 3 到 7 个更细步骤，再开始实际修改
6. 完成后回写状态、验证结论和交接建议

## 3. 全局共同行为

无论哪个 cluster，执行窗口都应遵守：

- 只围绕自己 cluster 的边界推进，不主动越界改其他 cluster 的核心实现
- 默认只处理“进行中 / 未完成”的子任务，已完成项不重做
- 如果需要跨 cluster 改动核心文件，先按 `Cluster-A` 口径判断
- 代码类任务必须补至少一轮相关验证
- 文档类任务必须补清入口、使用方式和交接建议
- 用户要求 `[log]` 时，任务完成后同步更新 `prompt-history.md`

## 4. 模块边界速查

| Cluster | 主要负责 | 当前重点 | 默认不要做 |
| --- | --- | --- | --- |
| A | 范围、状态、优先级、冲突、验收 | 状态板与阶段口径 | 直接替代实现窗口写大块业务代码 |
| B | contracts 与共享协议 | 字段说明、变更流程 | 扩业务逻辑 |
| C | 主 API 链路、provider、URL 输入 | `C9`、`C10` | 真实检索系统、前端页面 |
| D | 检索、去重、时间线、缓存 | 真实检索与真实时间线 | 前端组件与总控收口 |
| E | 前端工作台 | 默认只做残余联调/收口 | 重做已完成页面骨架 |
| F | eval 回归、smoke、验收 | `F2`、`F6`、`F7` | 长期代替实现窗口修业务 |
| G | demo、README、演示口径、replay 说明 | `G3`、`G4`、`G5`、`G6` | 直接扩主链路功能 |

## 5. 当前推荐派工顺序

1. `Cluster-D`
   优先补真实检索、标准化、时间线能力。
2. `Cluster-C`
   优先完成 URL 正文抽取与 fallback。
3. `Cluster-F`
   优先把输入、模式和 smoke checklist 建起来。
4. `Cluster-G`
   同步收口运行方式、限制说明和演示脚本。
5. `Cluster-A`
   持续维护状态板和阶段口径，避免再次失真。

## 6. 什么时候还需要额外 prompt

只有在以下情况，才建议另写更细 prompt：

- 某个 cluster 内部又出现明显独立且高风险的子方向
- 用户明确要求把一个 cluster 再拆成多个窗口
- 当前窗口在执行中遇到边界不清或需要做技术路线取舍

否则，优先使用每个 cluster 文件自带的“窗口执行 Prompt（全局）”。
