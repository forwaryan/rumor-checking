# `prompt-history.md` 全量归档：按核心链路整理

## 1. 这份归档在做什么

这份文档不是继续挑“好 prompt”，而是把 `prompt-history.md` 里出现过的全部记录，按照新闻核查产品的核心链路重排。

换句话说，这里回答的是：

- 这些 prompt 在整个产品设计过程中分别处于哪一段
- 每一段沉淀了什么类型的积累
- 哪些条目属于定义问题，哪些属于实现，哪些属于审计，哪些属于运行时问题回灌

说明：

- 本归档共覆盖 `39` 条记录。
- 个别原始条目存在混排或乱码，例如 `2026-03-13 23:24`、`2026-03-14 23:36 [log]`；这里按其“任务意图”归档，而不是按文本完整度归档。

```mermaid
flowchart LR
    A["A. 问题定义与目标对齐"] --> B["B. V1 边界与架构基线"]
    B --> C["C. 任务拆解与并行协作"]
    C --> D["D. 核心能力实现与模块沉淀"]
    D --> E["E. 状态审计、文档纠偏与能力评估"]
    E --> F["F. 运行时问题、失败样例与回归线索"]
    F --> G["G. 总控规划与最终执行板"]
    E -.回写任务与文档.-> C
    F -.失败样例反哺实现.-> D
```

## 5. 如果你要按核心链路讲给别人听，建议这样讲

### 5.1 产品链路版

先讲 `A + B`：

我们一开始不是直接写功能，而是先用 prompt 统一理解题目、竞品、评测和约束，再冻结 V1 边界和架构蓝图。

再讲 `C + D`：

然后我们把方案拆成可以并行推进的 cluster，再分别去做前端壳、真实 provider、真实 retrieval 和模块级文档沉淀。

最后讲 `E + F + G`：

中后期我们不断用 prompt 做状态审计、文档回写、失败样例回灌和运行时问题定位，最后再把这些积累收口成统一的执行母板和高分路线。

### 5.2 设计过程版

如果你想强调“prompt 不是一次性工具”，可以直接这样说：

这份 `prompt-history.md` 其实记录的是一个完整的设计闭环。前面几类 prompt 帮我定义问题和冻结边界，中间几类 prompt 帮我拆任务和落主链能力，后面几类 prompt 则负责校验真实性、收集失败样例、修正文档口径，最后再把这些积累升格成统一的总控方案。

## 6. 最后一句结论

如果按核心链路看，`prompt-history.md` 不是一串零散的 prompt，而是一套完整的项目积累系统：

> 从问题定义，到 V1 架构，到并行协作，到真实能力实现，到状态审计，到失败样例回灌，再到最终执行母板，它几乎把整个产品设计过程都记录成了可复用资产。
